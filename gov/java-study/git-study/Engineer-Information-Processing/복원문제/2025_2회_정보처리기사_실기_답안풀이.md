## 문제 1

**문제:** 데이터베이스 물리 설계 시 레코드 접근 방법에서 빈칸을 채우시오.
> 순차 접근, [    ], 해싱

### 답: 인덱스

### 풀이

데이터베이스 물리 설계에서 레코드에 접근하는 방법은 크게 세 가지로 분류된다.

1. **순차 접근(Sequential Access):** 레코드를 처음부터 순서대로 하나씩 탐색하는 방법이다.
2. **인덱스(Index):** 별도의 인덱스 테이블을 구성하여 원하는 레코드의 위치를 빠르게 찾아가는 방법이다. B-트리, B+트리 등의 자료구조가 대표적이다.
3. **해싱(Hashing):** 해시 함수를 사용하여 키 값을 주소로 변환해 직접 접근하는 방법이다.

인덱스는 순차 접근보다 빠르면서도 해싱처럼 충돌 문제가 발생하지 않아 가장 범용적으로 사용되는 접근 방법이다.

## 문제 2

**문제:** 릴레이션의 열(Column)을 설명하는 개념으로 "학번, 이름, 전공 등"의 속성을 나타내는 것을 보기에서 고르시오.

### 답: ㄷ. Attribute

### 풀이

관계형 데이터베이스에서 릴레이션 관련 용어를 정리하면 다음과 같다.

| 용어 | 설명 |
|------|------|
| **Attribute (속성)** | 릴레이션의 열(Column)을 의미하며, 학번, 이름, 전공 등 개별 항목을 나타낸다. |
| **Tuple (튜플)** | 릴레이션의 행(Row)을 의미하며, 하나의 레코드에 해당한다. |
| **Degree (차수)** | 릴레이션에 포함된 속성(Attribute)의 수를 의미한다. |
| **Cardinality (기수)** | 릴레이션에 포함된 튜플(Tuple)의 수를 의미한다. |
| **Domain (도메인)** | 하나의 속성이 가질 수 있는 값의 범위를 의미한다. |
| **Schema (스키마)** | 데이터베이스의 전체적인 구조와 제약 조건을 정의한 것이다. |

문제에서 "학번, 이름, 전공 등"은 릴레이션의 개별 열에 해당하므로 **Attribute**가 정답이다.

## 문제 3

**문제:** 포트 22번을 사용하며, 공개키 기반 인증과 암호화 통신을 제공하는 원격 접속 프로토콜의 이름을 쓰시오.

### 답: SSH

### 풀이

**SSH(Secure Shell)**는 네트워크를 통해 원격 시스템에 안전하게 접속하기 위한 프로토콜이다.

- **포트 번호:** 기본 포트 22번을 사용한다.
- **인증 방식:** 공개키(Public Key) 기반 인증을 지원하며, 비밀번호 인증도 가능하다.
- **암호화:** 통신 내용을 암호화하여 도청, 중간자 공격 등을 방지한다.
- **기존 Telnet(포트 23)의 대안:** Telnet은 평문(Plain Text)으로 통신하여 보안에 취약하지만, SSH는 모든 데이터를 암호화하여 전송한다.

SSH는 원격 서버 관리, 파일 전송(SCP, SFTP) 등에 광범위하게 사용된다.

## 문제 4

**문제:**
(1) CPU burst가 짧은 프로세스를 우선 처리하는 비선점형 스케줄링 알고리즘
(2) (1)의 선점형 구현 방식

### 답: (1) SJF, (2) SRT

### 풀이

**(1) SJF (Shortest Job First)**
- **비선점형** 스케줄링 알고리즘이다.
- CPU burst time(실행 시간)이 가장 짧은 프로세스에게 먼저 CPU를 할당한다.
- 일단 CPU를 할당받으면 해당 프로세스가 완료될 때까지 CPU를 빼앗지 않는다.
- 평균 대기시간을 최소화하는 최적의 알고리즘이지만, 실행 시간을 미리 알기 어렵다는 단점이 있다.
- 긴 작업이 계속 밀리는 **기아(Starvation)** 현상이 발생할 수 있다.

**(2) SRT (Shortest Remaining Time)**
- SJF의 **선점형** 버전이다.
- SRTF(Shortest Remaining Time First)라고도 한다.
- 현재 실행 중인 프로세스의 남은 실행 시간보다 더 짧은 실행 시간을 가진 프로세스가 도착하면, 현재 프로세스를 중단하고 새 프로세스에게 CPU를 할당한다.
- 선점이 가능하므로 SJF보다 평균 대기시간이 더 짧지만, 잦은 문맥 교환(Context Switching)으로 오버헤드가 발생할 수 있다.

## 문제 5

**문제:** 다음 Java 코드의 출력값을 쓰시오.

```java
public class Main {
    public static void change(String[] data, String s){
        data[0] = s;
        s = "Z";
    }
    public static void main(String[] args) {
        String data[] = { "A" };
        String s = "B";
        change(data, s);
        System.out.print(data[0] + s);
    }
}
```

### 답: BB

### 풀이

이 문제는 Java의 **매개변수 전달 방식**에 대한 이해를 묻는 문제이다.

**실행 흐름:**

1. `main`에서 `data = {"A"}`, `s = "B"` 선언
2. `change(data, s)` 호출
   - `data`는 배열(참조 타입)이므로 **배열 객체의 참조**가 전달된다. 따라서 `change` 메서드 내에서 `data[0]`을 변경하면 원본 배열에도 영향을 준다.
   - `s`는 String 타입으로 참조가 복사되어 전달된다.
3. `change` 메서드 내부:
   - `data[0] = s;` → `data[0]`이 `"B"`로 변경됨 (원본 배열 영향 O)
   - `s = "Z";` → 메서드 내부의 지역 변수 `s`만 `"Z"`로 변경됨 (main의 `s`에 영향 X)
4. `change` 메서드 종료 후 `main`으로 복귀:
   - `data[0]` = `"B"` (변경됨)
   - `s` = `"B"` (변경되지 않음)
5. 출력: `"B" + "B"` = **BB**

**핵심 포인트:** Java에서 배열은 참조 타입이므로 메서드 내에서 배열 원소를 변경하면 원본에 반영된다. 반면 String 변수에 새 값을 할당하면 지역 변수의 참조만 바뀌고 호출자의 변수에는 영향이 없다.

## 문제 6

**문제:** IP 주소가 223.13.234.132이고, 서브넷 마스크가 255.255.255.192일 때
① 네트워크 주소의 마지막 옥텟
② 할당 가능한 호스트 수

### 답: ① 128, ② 62

### 풀이

**서브넷 마스크 분석:**
- 서브넷 마스크: 255.255.255.192
- 마지막 옥텟 192를 이진수로 변환: `11000000`
- 서브넷 비트: 상위 2비트 → 호스트 비트: 하위 6비트

**① 네트워크 주소의 마지막 옥텟:**
- IP의 마지막 옥텟: 132 = `10000100`
- 서브넷 마스크의 마지막 옥텟: 192 = `11000000`
- AND 연산: `10000100` AND `11000000` = `10000000` = **128**

**② 할당 가능한 호스트 수:**
- 호스트 비트 수: 6비트
- 전체 주소 수: 2^6 = 64개
- 네트워크 주소(128)와 브로드캐스트 주소(191)를 제외: 64 - 2 = **62개**

**서브넷 정리:**
- 네트워크 주소: 223.13.234.128
- 브로드캐스트 주소: 223.13.234.191
- 할당 가능 범위: 223.13.234.129 ~ 223.13.234.190 (총 62개)

## 문제 7

**문제:** 객체에 대한 접근을 제어하기 위해 대리 객체를 사용하며, 지연 생성을 통해 메모리를 절약할 수 있는 디자인 패턴의 이름을 쓰시오.

### 답: Proxy

### 풀이

**Proxy 패턴**은 GoF 디자인 패턴 중 **구조 패턴(Structural Pattern)**에 해당한다.

**개념:**
- 실제 객체에 대한 접근을 제어하기 위해 **대리 객체(Proxy)**를 두는 패턴이다.
- 클라이언트는 실제 객체 대신 프록시 객체를 통해 간접적으로 접근한다.

**Proxy 패턴의 종류:**
| 종류 | 설명 |
|------|------|
| **가상 프록시(Virtual Proxy)** | 생성 비용이 큰 객체를 실제로 필요할 때까지 생성을 지연시켜 메모리를 절약한다. |
| **보호 프록시(Protection Proxy)** | 객체에 대한 접근 권한을 제어한다. |
| **원격 프록시(Remote Proxy)** | 원격 서버에 있는 객체를 로컬에서 대리한다. |

문제에서 "대리 객체 사용", "지연 생성", "메모리 절약" 등의 키워드가 모두 Proxy 패턴의 특징이다.

## 문제 8

**문제:** JavaScript와 XML/JSON을 이용하여 비동기적으로 웹 페이지를 갱신하고, 동적인 기능을 구현하는 기술의 이름을 쓰시오.

### 답: AJAX

### 풀이

**AJAX (Asynchronous JavaScript and XML)**는 웹 페이지 전체를 새로고침하지 않고도 서버와 비동기적으로 데이터를 주고받아 화면의 일부분만 동적으로 갱신하는 기술이다.

**핵심 특징:**
- **비동기 통신:** 서버와 비동기적으로 데이터를 교환하므로, 사용자는 페이지 전체가 리로드될 때까지 기다릴 필요가 없다.
- **부분 갱신:** 전체 페이지가 아닌 필요한 부분만 갱신하여 사용자 경험(UX)을 향상시킨다.
- **사용 기술:** JavaScript, XMLHttpRequest(또는 Fetch API), XML/JSON, HTML, CSS 등을 조합하여 사용한다.

**동작 원리:**
1. 사용자의 이벤트 발생 (클릭, 입력 등)
2. JavaScript가 XMLHttpRequest 객체를 생성하여 서버에 비동기 요청
3. 서버가 요청을 처리하고 XML/JSON 형태로 응답
4. JavaScript가 응답 데이터를 받아 DOM을 조작하여 화면 일부를 갱신

## 문제 9

**문제:** 다음 Java 코드의 출력값을 쓰시오.

```java
static interface F {
    int apply(int x) throws Exception;
}

static int run(F f) {
    try {
        return f.apply(2);
    } catch (Exception e) {
        return 10;
    }
}

public static void main(String[] args) {
    F f = (x) -> {
        if (x > 2) throw new Exception();
        return x * 2;
    };
    System.out.print(run(f) + run((int n) -> n + 9));
}
```

### 답: 19

### 풀이

이 문제는 Java의 **함수형 인터페이스**와 **람다식**, 그리고 **예외 처리**를 다룬다.

**실행 흐름:**

**1단계: `run(f)` 호출**
- `f.apply(2)` 실행
- 람다식 내부: `x = 2`, `if (2 > 2)` → **false** → 예외 발생하지 않음
- `return x * 2` = `return 2 * 2` = **4**

**2단계: `run((int n) -> n + 9)` 호출**
- 익명 람다식 `(int n) -> n + 9`이 전달됨
- `f.apply(2)` 실행 → `n = 2`, `return 2 + 9` = **11**

**3단계: 결과 합산**
- `run(f) + run(lambda)` = 4 + 11 = **15**

> **참고:** 코드를 그대로 실행하면 15가 출력되지만, 복원 문제의 원본 답안은 **19**로 기재되어 있다. 이는 실제 시험에서의 코드가 복원 과정에서 일부 달라졌을 가능성이 있다. (예: `x >= 2` 조건이었다면 `run(f)`에서 예외 발생 → `return 10`, 그리고 `10 + 9 = 19`가 된다.) 시험 원본 기준 답은 **19**이다.

## 문제 10

**문제:** 다음 Java 코드의 출력값을 쓰시오.

```java
class Parent {
    String id() { return "P"; }
    int x(int a) { return a * 2; }
}

class Child extends Parent {
    String id() { return "C"; }
    int x(int a) { return a + 3; }
}

public class Main {
    public static void main(String[] args) {
        Parent ref = new Child();
        System.out.println(ref.x(2) + ref.id());
    }
}
```

### 답: 5C

### 풀이

이 문제는 Java의 **다형성(Polymorphism)**과 **동적 바인딩(Dynamic Binding)**을 묻는 문제이다.

**핵심 개념:**
- `Parent ref = new Child();` → 참조 변수는 `Parent` 타입이지만, 실제 객체는 `Child` 인스턴스이다.
- 메서드 호출 시 **런타임에 실제 객체의 타입(Child)**에 따라 오버라이딩된 메서드가 호출된다.

**실행 흐름:**

1. `ref.x(2)` 호출
   - `ref`의 실제 객체는 `Child` → `Child`의 `x(int a)` 실행
   - `return 2 + 3` = **5**

2. `ref.id()` 호출
   - `ref`의 실제 객체는 `Child` → `Child`의 `id()` 실행
   - `return "C"`

3. `ref.x(2) + ref.id()` = `5 + "C"`
   - Java에서 `int + String`은 문자열 결합 → **"5C"**

4. 출력: **5C**

> 참고: 문제 원본에서 `id()` 메서드가 `"P"`를 반환하는 경우(즉, Parent의 메서드가 호출되는 경우) 답이 **5P**가 될 수도 있다. 그러나 다형성 원리에 따르면 실제 객체인 Child의 메서드가 호출되므로 **5C**가 올바른 답이다. 복원 문제에 따라 답이 5P로 기재된 경우도 있으나, 코드 기준으로는 **5C**이다.

## 문제 11

**문제:** 제어 흐름 그래프에서 분기 커버리지(Branch Coverage)를 만족하는 테스트 경로를 쓰시오.

### 답: 1234561, 124567 (또는 1234567, 124561)

### 풀이

**분기 커버리지(Branch Coverage = 결정 커버리지)**는 프로그램의 모든 분기(조건의 참/거짓)를 최소 한 번씩 실행하는 테스트 기법이다.

**분기 커버리지의 조건:**
- 모든 조건문의 **참(True)** 경로와 **거짓(False)** 경로를 최소 한 번씩 수행해야 한다.
- 제어 흐름 그래프의 모든 간선(Edge)을 최소 한 번씩 통과해야 한다.

**경로 해석:**
- **경로 1 (1→2→3→4→5→6→1):** 첫 번째 조건 참, 두 번째 조건 참 경로
- **경로 2 (1→2→4→5→6→7):** 첫 번째 조건 거짓, 두 번째 조건 거짓 경로

이 두 경로를 조합하면 모든 분기의 참/거짓을 최소 한 번씩 통과하므로 분기 커버리지를 만족한다.

## 문제 12

**문제:** 다음 C 언어 큐(Queue) 코드의 실행 결과를 쓰시오.

```c
enq(&q, 1);
enq(&q, 2);
deq(&q);
enq(&q, 3);
// 이후 남은 큐 원소 출력
```

### 답: 2, 3

### 풀이

**큐(Queue)**는 **FIFO(First In, First Out)** 구조의 자료구조이다. 먼저 들어온 데이터가 먼저 나간다.

**실행 흐름:**

| 연산 | 큐 상태 (front → rear) | 설명 |
|------|------------------------|------|
| `enq(&q, 1)` | [1] | 1을 큐에 삽입 |
| `enq(&q, 2)` | [1, 2] | 2를 큐에 삽입 |
| `deq(&q)` | [2] | front에서 1을 제거 (FIFO) |
| `enq(&q, 3)` | [2, 3] | 3을 큐에 삽입 |

**최종 큐 상태:** [2, 3]

따라서 남은 원소를 순서대로 출력하면 **2, 3**이다.

## 문제 13

**문제:** 라운드 로빈(Round Robin) 스케줄링에서 타임 퀀텀이 4ms일 때 평균 대기시간을 구하시오.

| 프로세스 | 도착 시간 | 실행 시간 |
|---------|----------|----------|
| P1      | 0ms      | 8ms      |
| P2      | 1ms      | 4ms      |
| P3      | 2ms      | 9ms      |
| P4      | 3ms      | 5ms      |

### 답: 11.75ms

### 풀이

**라운드 로빈(Round Robin)**은 각 프로세스에 동일한 시간(타임 퀀텀)만큼 CPU를 할당하고, 시간이 초과되면 준비 큐의 맨 뒤로 보내는 선점형 스케줄링이다.

**실행 과정 (타임 퀀텀 = 4ms):**

| 시간 구간 | 실행 프로세스 | 남은 실행 시간 | 대기 큐 상태 |
|-----------|-------------|--------------|-------------|
| 0~4ms | P1 (4ms 실행) | P1:4 | P2, P3, P4 도착 |
| 4~8ms | P2 (4ms 실행) | P2:0 (완료) | P3, P4, P1 |
| 8~12ms | P3 (4ms 실행) | P3:5 | P4, P1 |
| 12~16ms | P4 (4ms 실행) | P4:1 | P1, P3 |
| 16~20ms | P1 (4ms 실행) | P1:0 (완료) | P3, P4 |
| 20~24ms | P3 (4ms 실행) | P3:1 | P4 |
| 24~25ms | P4 (1ms 실행) | P4:0 (완료) | P3 |
| 25~26ms | P3 (1ms 실행) | P3:0 (완료) | - |

**각 프로세스의 대기시간 계산:**
- 대기시간 = 완료 시간 - 도착 시간 - 실행 시간

| 프로세스 | 완료 시간 | 도착 시간 | 실행 시간 | 대기 시간 |
|---------|----------|----------|----------|----------|
| P1 | 20ms | 0ms | 8ms | 20 - 0 - 8 = **12ms** |
| P2 | 8ms | 1ms | 4ms | 8 - 1 - 4 = **3ms** |
| P3 | 26ms | 2ms | 9ms | 26 - 2 - 9 = **15ms** |
| P4 | 25ms | 3ms | 5ms | 25 - 3 - 5 = **17ms** |

**평균 대기시간:** (12 + 3 + 15 + 17) / 4 = 47 / 4 = **11.75ms**

## 문제 14

**문제:** 다음 C 언어 코드의 출력값을 쓰시오.

```c
struct dat { int x; int y; };
struct dat a[] = {{1, 2}, {3, 4}, {5, 6}};
struct dat (*pptr)[3] = &a;

(*pptr)[1] = (*pptr)[2];
printf("%d %d", (*pptr)[1].x, (*pptr)[1].y);
```

### 답: 5 6

### 풀이

이 문제는 C 언어의 **구조체 배열**과 **배열 포인터**에 대한 이해를 묻는다.

**코드 분석:**

1. `struct dat a[]` 선언:
   - `a[0]` = {1, 2}
   - `a[1]` = {3, 4}
   - `a[2]` = {5, 6}

2. `struct dat (*pptr)[3] = &a;`
   - `pptr`는 "크기 3인 struct dat 배열"을 가리키는 포인터이다.
   - `&a`는 배열 전체의 주소를 의미한다.
   - `*pptr`은 배열 `a` 자체를 의미하므로, `(*pptr)[i]`는 `a[i]`와 동일하다.

3. `(*pptr)[1] = (*pptr)[2];`
   - `a[1] = a[2]`와 동일
   - `a[1]`이 {5, 6}으로 변경됨 (구조체 대입은 값 전체 복사)

4. `printf("%d %d", (*pptr)[1].x, (*pptr)[1].y);`
   - `a[1].x` = **5**, `a[1].y` = **6**

**출력:** **5 6**

## 문제 15

**문제:** 다음 Java 코드의 출력값을 쓰시오.

```java
// 배열에 객체 저장 후 참조 교환
arr[0] = arr[2];
arr[2] = t;
arr[1].v = arr[0].v;
// 출력
```

### 답: 1a3b3

### 풀이

이 문제는 Java의 **객체 참조(Reference)**와 **배열에서의 참조 교환**을 다루는 문제이다.

**핵심 개념:**
- Java에서 배열에 객체를 저장하면, 배열 원소는 객체 자체가 아니라 **객체의 참조(주소)**를 저장한다.
- 참조를 교환하면 같은 객체를 여러 변수가 가리킬 수 있다.
- 한 참조를 통해 객체의 값을 변경하면, 같은 객체를 가리키는 다른 참조에서도 변경된 값이 보인다.

**실행 결과:** 참조 교환과 값 대입의 순서에 따라 최종 출력이 **1a3b3**이 된다.

## 문제 16

**문제:** 다음 C 언어 연결리스트 코드의 출력값을 쓰시오.

```c
// 노드 a(1), b(2), c(3) 생성
c.n = &a;
a.n = &b;
b.n = NULL;
// 순회하며 출력
```

### 답: 3 1 2

### 풀이

이 문제는 C 언어의 **연결 리스트(Linked List)** 구조를 이해하는 문제이다.

**노드 생성:**
- 노드 a: 데이터 = 1
- 노드 b: 데이터 = 2
- 노드 c: 데이터 = 3

**연결 설정:**
```
c.n = &a;   →  c(3) → a
a.n = &b;   →  a(1) → b
b.n = NULL; →  b(2) → NULL (끝)
```

**연결 리스트 구조:**
```
c(3) → a(1) → b(2) → NULL
```

**순회 출력:**
- 시작 노드: c → 데이터 **3** 출력
- 다음 노드: a → 데이터 **1** 출력
- 다음 노드: b → 데이터 **2** 출력
- 다음 노드: NULL → 종료

**출력:** **3 1 2**

## 문제 17

**문제:** 다음 Python 코드의 출력값을 쓰시오.

```python
lst = [1, 2, 3, 4, 5]
dst = {i : i * 2 for i in lst}
result = s & set(dst.values())
print(len(result))
```

### 답: 2

### 풀이

**코드 분석:**

1. `lst = [1, 2, 3, 4, 5]`

2. `dst = {i : i * 2 for i in lst}` → 딕셔너리 컴프리헨션
   - `dst = {1: 2, 2: 4, 3: 6, 4: 8, 5: 10}`

3. `dst.values()` = `[2, 4, 6, 8, 10]`
   - `set(dst.values())` = `{2, 4, 6, 8, 10}`

4. `result = s & set(dst.values())`
   - `&`는 집합의 **교집합(intersection)** 연산이다.
   - `s`는 특정 set이며, `{2, 4, 6, 8, 10}`과의 교집합 원소 개수가 2개가 되도록 정의되어 있다.

5. `len(result)` = **2**

**핵심 개념:**
- **딕셔너리 컴프리헨션:** `{key: value for 변수 in 반복가능객체}` 형태로 딕셔너리를 간결하게 생성
- **집합 연산:** `&`(교집합), `|`(합집합), `-`(차집합), `^`(대칭차집합)

## 문제 18

**문제:** 다음 C 언어 함수의 실행 결과를 쓰시오.

```c
// 동적 메모리 연결리스트를 이용한 문자열 역순 출력
func("BEST");
```

### 답: TSEB

### 풀이

이 문제는 **연결 리스트**를 사용한 **문자열 역순 출력** 알고리즘이다.

**동작 원리:**

1. 문자열 "BEST"의 각 문자를 하나씩 연결 리스트의 **앞(head)**에 삽입한다.
   - 'B' 삽입: B → NULL
   - 'E' 삽입: E → B → NULL
   - 'S' 삽입: S → E → B → NULL
   - 'T' 삽입: T → S → E → B → NULL

2. 연결 리스트를 순회하며 출력한다.
   - T → S → E → B

**이것은 사실상 스택(Stack)과 동일한 원리이다:**
- 연결 리스트의 앞에 삽입하는 것은 스택의 push와 같다.
- 순회 출력은 스택의 pop 순서와 같다.
- **LIFO(Last In, First Out)** 구조이므로 마지막에 삽입된 'T'가 먼저 출력된다.

**출력:** **TSEB**

## 문제 19

**문제:** TCP 3-way-handshake 과정 중 SYN 패킷만 보내고 ACK를 보내지 않아 서버의 자원을 소모시키는 공격 기법의 이름을 쓰시오.

### 답: SYN Flooding

### 풀이

**SYN Flooding(SYN 플러딩)**은 **DoS(Denial of Service)** 공격의 대표적인 유형이다.

**정상적인 TCP 3-way-handshake:**
1. 클라이언트 → 서버: **SYN** (연결 요청)
2. 서버 → 클라이언트: **SYN + ACK** (요청 수락)
3. 클라이언트 → 서버: **ACK** (연결 확립)

**SYN Flooding 공격 과정:**
1. 공격자가 대량의 **SYN** 패킷을 서버에 전송 (출발지 IP를 위조)
2. 서버는 각 SYN에 대해 **SYN + ACK**를 응답하고, 연결을 위한 자원(백로그 큐)을 할당하며 ACK를 대기
3. 공격자는 **ACK를 보내지 않음** (IP가 위조되었으므로 응답 불가)
4. 서버의 백로그 큐가 가득 차서 정상적인 연결 요청을 처리할 수 없게 됨

**대응 방법:**
- **SYN Cookie:** 백로그 큐를 사용하지 않고 SYN-ACK에 쿠키 값을 포함시켜 ACK 수신 시 검증
- **방화벽/IDS 설정:** 비정상적인 SYN 패킷 탐지 및 차단
- **백로그 큐 크기 증가:** 임시 대응책
- **타임아웃 단축:** Half-open 연결의 대기 시간 축소

## 문제 20

**문제:** 다음 관계대수식의 연산 결과를 쓰시오.

```
πTTL(employee)
```

### 답: 부장, 대리, 과장, 차장

### 풀이

**관계대수(Relational Algebra)**의 **프로젝션(Projection, π)** 연산에 대한 문제이다.

**프로젝션(π) 연산:**
- 릴레이션에서 **특정 속성(열)**만 선택하여 추출하는 연산이다.
- 수학적 표기: **π속성명(릴레이션명)**
- SQL에서는 `SELECT 속성명 FROM 릴레이션명`에 해당한다.
- **중복 제거:** 프로젝션 결과에서 중복된 튜플은 자동으로 제거된다.

**πTTL(employee) 해석:**
- `employee` 릴레이션에서 `TTL`(직급) 속성만 추출한다.
- 중복을 제거한 결과: **부장, 대리, 과장, 차장**

**관계대수 주요 연산 정리:**

| 연산 | 기호 | 설명 |
|------|------|------|
| 셀렉션 | σ (시그마) | 조건을 만족하는 **행(튜플)** 선택 |
| 프로젝션 | π (파이) | 특정 **열(속성)** 선택 |
| 조인 | ⋈ | 두 릴레이션을 공통 속성으로 결합 |
| 디비전 | ÷ | 한 릴레이션의 모든 튜플과 연관된 튜플 추출 |
| 합집합 | ∪ | 두 릴레이션의 합 |
| 차집합 | - | 한 릴레이션에만 있는 튜플 |
| 교집합 | ∩ | 두 릴레이션에 공통으로 있는 튜플 |
| 카티션 프로덕트 | × | 두 릴레이션의 모든 조합 |
