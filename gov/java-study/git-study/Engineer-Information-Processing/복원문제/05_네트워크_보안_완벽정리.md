## 1.2 SSH (Secure Shell) [빈출]

```
[핵심 정리]
- 포트 번호: 22번
- 인증 방식: 공개키(비대칭키) 기반
- 기능: 암호화된 원격 접속 (Telnet의 보안 버전)
- 대체 대상: Telnet(포트 23, 평문 전송) -> SSH(암호화 전송)
```

**시험에 이렇게 나온다:**
> Q. 포트 22번을 사용하며 공개키 기반의 암호화된 원격 접속 프로토콜은?
> A. **SSH (Secure Shell)**

**Telnet과 SSH 비교:**

| 구분 | Telnet | SSH |
|------|--------|-----|
| 포트 | 23 | 22 |
| 보안 | 평문 전송 (보안 취약) | 암호화 전송 |
| 인증 | 패스워드만 | 공개키 + 패스워드 |

**암기법:** "SSH = **S**ecure **Sh**ell = **안전한 껍데기** = 포트 **22**(이이) = 공개키"

## 1.3 ARP / RARP [최빈출]

```
[핵심 정리]
ARP  (Address Resolution Protocol)  : IP 주소  -> MAC 주소 변환
RARP (Reverse ARP)                  : MAC 주소 -> IP 주소 변환
```

**동작 원리:**
```
ARP 동작 과정:
1. "IP 192.168.1.5의 MAC 주소가 뭐지?" (브로드캐스트)
2. 해당 IP를 가진 호스트가 자기 MAC 주소를 응답 (유니캐스트)
3. ARP 캐시 테이블에 저장

RARP 동작 과정:
1. "내 MAC 주소는 AA:BB:CC:DD인데, 내 IP가 뭐지?" (브로드캐스트)
2. RARP 서버가 해당 MAC에 대응하는 IP 주소를 응답
```

**암기법:**
- **ARP** = "**A**ddress를 **R**esolve(해결)" = IP를 알고 MAC을 모를 때 -> **IP에서 MAC으로**
- **RARP** = "**R**everse(반대)" = ARP의 반대 = **MAC에서 IP로**
- 핵심: **A**RP = **A**sk MAC(맥을 물어봄) / **R**ARP = **R**everse(반대로)

**시험에 이렇게 나온다:**
> Q. IP 주소를 물리적 주소(MAC)로 변환하는 프로토콜은?
> A. **ARP**

> Q. 물리적 주소(MAC)를 IP 주소로 변환하는 프로토콜은?
> A. **RARP**

## 1.4 ICMP (Internet Control Message Protocol)

```
[핵심 정리]
- 인터넷 제어 메시지 프로토콜
- 기능: 오류 보고(Error Reporting), 상태 진단
- 대표 명령어: ping (목적지까지 도달 가능한지 확인)
- 계층: 네트워크 계층 (3계층)
- IP 프로토콜의 비신뢰성을 보완
```

**ICMP 메시지 유형:**

| 유형 | 설명 |
|------|------|
| Echo Request/Reply | ping 요청/응답 |
| Destination Unreachable | 목적지 도달 불가 |
| Time Exceeded | TTL 초과 (traceroute가 사용) |
| Redirect | 라우팅 경로 변경 알림 |

**암기법:** "ICMP = **I**nternet **C**ontrol **M**essage = 인터넷 **제어 메시지** = **오류 보고** = **ping**"

**주의:** ICMP는 보안 공격에도 악용됨!
- 스머프(Smurf) 공격: ICMP Echo Reply 대량 발생
- ICMP Flooding: ping 연속 전송 (= Ping of Death)

## 1.5 라우팅 프로토콜: OSPF vs RIP [빈출]

### 라우팅 프로토콜 분류

```
라우팅 프로토콜
├── IGP (Interior Gateway Protocol) - 내부 게이트웨이
│   ├── 거리 벡터 방식: RIP
│   └── 링크 상태 방식: OSPF
└── EGP (Exterior Gateway Protocol) - 외부 게이트웨이
    └── BGP
```

### OSPF vs RIP 완벽 비교

| 구분 | RIP | OSPF |
|------|-----|------|
| **알고리즘** | 거리 벡터 (Distance Vector) | 링크 상태 (Link State) |
| **메트릭** | Hop Count (홉 수) | 대역폭(Bandwidth) 기반 비용 |
| **최대 홉** | **15홉** (16홉은 도달 불가) | 제한 없음 |
| **라우팅 알고리즘** | Bellman-Ford | **Dijkstra (최단 경로)** |
| **업데이트 주기** | 30초마다 전체 라우팅 테이블 | 변화 있을 때만 (빠른 수렴) |
| **네트워크 규모** | 소규모 | 대규모 |
| **프로토콜 유형** | IGP | IGP |
| **특징** | 구현 간단, 오버헤드 적음 | 빠른 수렴, 계층적 구조 |

**암기법:**
- **RIP** = "**R**outing **I**nformation **P**rotocol" = **립**다리 **거리** = 거리 벡터 = **Hop Count** = 최대 **15**홉
- **OSPF** = "**O**pen **S**hortest **P**ath **F**irst" = **최단 경로** 우선 = **Dijkstra** = 링크 상태

**시험에 이렇게 나온다:**
> Q. 링크 상태 라우팅 알고리즘으로 Dijkstra 알고리즘을 사용하는 내부 게이트웨이 프로토콜은?
> A. **OSPF**

> Q. 거리 벡터 라우팅 프로토콜로 최대 홉 수가 15인 프로토콜은?
> A. **RIP**

## 1.6 IPSec (IP Security) [빈출]

```
[핵심 정리]
- 계층: 네트워크 계층 (3계층)
- 기능: IP 패킷을 암호화하여 안전하게 전송
- 구성 프로토콜: AH + ESP
- VPN 구현의 핵심 기술
```

### IPSec 구성 프로토콜

| 구성요소 | 기능 | 제공 서비스 |
|----------|------|-------------|
| **AH** (Authentication Header) | 인증 + 무결성 | 데이터 변조 방지 (암호화 X) |
| **ESP** (Encapsulating Security Payload) | 인증 + 무결성 + **기밀성** | 데이터 암호화 포함 |
| **IKE** (Internet Key Exchange) | 키 교환 | 보안 연결(SA) 설정 |

### IPSec 동작 모드

| 모드 | 설명 |
|------|------|
| **전송 모드** (Transport) | IP 페이로드만 암호화 (헤더는 그대로) |
| **터널 모드** (Tunnel) | IP 패킷 전체를 암호화 (새 IP 헤더 추가) |

**암기법:**
- IPSec = **IP** 패킷을 **Sec**ure(안전하게) = IP 암호화
- **AH** = **A**uthentication(인증) **H**eader = 인증만! 암호화 안 함
- **ESP** = **E**ncapsulating **S**ecurity **P**ayload = 암호화 포함! (ESP의 E = Encrypt로 연상)

## 1.7 기타 프로토콜 (L2TP, AJAX, ATM)

### L2TP (Layer 2 Tunneling Protocol)

```
[핵심 정리]
- 계층: 데이터 링크 계층 (2계층)
- 기능: 계층 2에서 터널링
- 탄생: L2F(Cisco) + PPTP(Microsoft) = L2TP
- VPN 구현에 사용
- 자체 암호화 없음 -> IPSec과 결합하여 사용
```

**암기법:** "L2TP = **L**ayer **2** **T**unneling = 2계층 터널 = L2F + PPTP **결합**"

### AJAX (Asynchronous JavaScript and XML)

```
[핵심 정리]
- 풀네임: Asynchronous JavaScript and XML
- 핵심: 비동기(Asynchronous) 통신
- 기능: 페이지 전체를 새로고침하지 않고 일부만 업데이트
- 구성: JavaScript + XML (최근에는 JSON도 사용)
```

**암기법:** "AJAX = **A**synchronous(비동기) + **J**avaScript + **X**ML = 비동기 웹 통신"

### ATM (Asynchronous Transfer Mode)

```
[핵심 정리]
- 핵심: 53바이트 고정 길이 셀(Cell) 단위 전송
- 셀 구조: 헤더 5바이트 + 데이터 48바이트 = 53바이트
- 특징: 실시간 데이터 전송에 적합 (음성, 영상)
- 교환 방식: 가상 회선 방식
```

**암기법:** "ATM = **53**바이트 = **오삼**(5+48) = 고정 **셀**"

# 2장. IP 주소와 서브넷 계산 [최빈출]

## 2.1 IP 주소 클래스

### IPv4 주소 체계

```
IPv4 주소 = 32비트 = 4개의 옥텟(Octet)
예: 192.168.1.100 = 11000000.10101000.00000001.01100100
```

### IP 클래스 구분

| 클래스 | 첫 번째 옥텟 범위 | 네트워크 비트 | 호스트 비트 | 기본 서브넷 마스크 | 용도 |
|--------|-------------------|---------------|-------------|-------------------|------|
| **A** | 0 ~ 127 | 8비트 | 24비트 | 255.0.0.0 (/8) | 대규모 네트워크 |
| **B** | 128 ~ 191 | 16비트 | 16비트 | 255.255.0.0 (/16) | 중규모 네트워크 |
| **C** | 192 ~ 223 | 24비트 | 8비트 | 255.255.255.0 (/24) | 소규모 네트워크 |
| **D** | 224 ~ 239 | - | - | - | 멀티캐스트 |
| **E** | 240 ~ 255 | - | - | - | 예약(연구용) |

**암기법:** "A(0~127) B(128~191) C(192~223) = **공일이 일이팔 일구이**"

**특수 IP 주소:**
- 127.0.0.1 : 루프백(Loopback) 주소 (자기 자신)
- 0.0.0.0 : 기본 경로 (Default Route)
- 255.255.255.255 : 제한된 브로드캐스트

## 2.2 서브넷 마스크 이해

### 서브넷 마스크란?

```
서브넷 마스크 = IP 주소에서 네트워크 부분과 호스트 부분을 구분하는 값

예: 255.255.255.0
    11111111.11111111.11111111.00000000
    ├──── 네트워크 부분 ────┤├─ 호스트 ─┤
```

### CIDR 표기법과 서브넷 마스크 대응

| CIDR | 서브넷 마스크 | 호스트 비트 | 호스트 수 |
|------|---------------|-------------|-----------|
| /8 | 255.0.0.0 | 24비트 | 16,777,214 |
| /16 | 255.255.0.0 | 16비트 | 65,534 |
| /24 | 255.255.255.0 | 8비트 | 254 |
| /25 | 255.255.255.128 | 7비트 | 126 |
| /26 | 255.255.255.192 | 6비트 | 62 |
| /27 | 255.255.255.224 | 5비트 | 30 |
| /28 | 255.255.255.240 | 4비트 | 14 |
| /29 | 255.255.255.248 | 3비트 | 6 |
| /30 | 255.255.255.252 | 2비트 | 2 |

**서브넷 마스크 값 외우기 (마지막 옥텟):**
```
/25 = 128  (10000000)
/26 = 192  (11000000)
/27 = 224  (11100000)
/28 = 240  (11110000)
/29 = 248  (11111000)
/30 = 252  (11111100)

규칙: 128, 192, 224, 240, 248, 252
      128 + 64 = 192 + 32 = 224 + 16 = 240 + 8 = 248 + 4 = 252
```

## 2.3 네트워크 주소 구하기 (AND 연산)

### 핵심 공식

```
네트워크 주소 = IP 주소 AND 서브넷 마스크
```

### 계산 방법

```
예제: IP 192.168.10.130, 서브넷 마스크 255.255.255.192 (/26)

  192.168.10.130  = 11000000.10101000.00001010.10 | 000010
  255.255.255.192 = 11111111.11111111.11111111.11 | 000000
  ─────────────────────────────────────────────────────────
  AND 결과        = 11000000.10101000.00001010.10 | 000000
                  = 192.168.10.128

네트워크 주소 = 192.168.10.128
```

**속산법:** 호스트 부분을 모두 0으로!

## 2.4 브로드캐스트 주소 구하기

### 핵심 공식

```
브로드캐스트 주소 = 네트워크 주소의 호스트 비트를 모두 1로
```

### 계산 방법

```
예제: 네트워크 주소 192.168.10.128/26

  네트워크 주소 = 192.168.10.128 = 11000000.10101000.00001010.10 | 000000
  호스트 비트를 모두 1로         = 11000000.10101000.00001010.10 | 111111
                                = 192.168.10.191

브로드캐스트 주소 = 192.168.10.191
```

**속산법:** 네트워크 주소 + (서브넷 크기 - 1)
- 서브넷 크기 = 256 - 서브넷 마스크 마지막 옥텟 = 256 - 192 = 64
- 브로드캐스트 = 128 + 64 - 1 = **191**

---

## 2.5 할당 가능 호스트 수

### 핵심 공식

```
할당 가능 호스트 수 = 2^(호스트 비트 수) - 2

-2 하는 이유:
  - 네트워크 주소 (호스트 비트 모두 0) -> 사용 불가
  - 브로드캐스트 주소 (호스트 비트 모두 1) -> 사용 불가
```

### 예시

| 서브넷 마스크 | 호스트 비트 | 계산 | 할당 가능 호스트 |
|---------------|-------------|------|-----------------|
| /24 (255.255.255.0) | 8비트 | 2^8 - 2 | **254개** |
| /25 (255.255.255.128) | 7비트 | 2^7 - 2 | **126개** |
| /26 (255.255.255.192) | 6비트 | 2^6 - 2 | **62개** |
| /27 (255.255.255.224) | 5비트 | 2^5 - 2 | **30개** |
| /28 (255.255.255.240) | 4비트 | 2^4 - 2 | **14개** |
| /29 (255.255.255.248) | 3비트 | 2^3 - 2 | **6개** |
| /30 (255.255.255.252) | 2비트 | 2^2 - 2 | **2개** |

## 2.6 같은 네트워크 판별법

```
두 IP 주소가 같은 네트워크에 있는지 확인:
  1. 각 IP를 서브넷 마스크와 AND 연산
  2. 결과(네트워크 주소)가 같으면 -> 같은 네트워크!
```

**예제:**
```
IP1: 192.168.1.100
IP2: 192.168.1.200
서브넷: 255.255.255.192 (/26)

IP1 AND 서브넷 = 192.168.1.64   (100 AND 192 = 01100100 AND 11000000 = 01000000 = 64)
IP2 AND 서브넷 = 192.168.1.192  (200 AND 192 = 11001000 AND 11000000 = 11000000 = 192)

64 =/= 192 -> 다른 네트워크!
```

## 2.7 서브넷 계산 3분 속산법 (실전 예제 5개)

### 속산법 정리

```
[3단계 속산법]
1단계: 호스트 비트 수 확인 (32 - CIDR 값)
2단계: 서브넷 블록 크기 = 2^(호스트 비트 수) 또는 256 - 마스크 마지막 옥텟
3단계: 네트워크 주소 = IP의 마지막 옥텟을 블록 크기로 나눈 몫 x 블록 크기
```

### 예제 1: 192.168.1.130/25

```
1단계: 호스트 비트 = 32 - 25 = 7비트
2단계: 블록 크기 = 2^7 = 128 (또는 256 - 128 = 128)
3단계: 130 / 128 = 1...2 -> 1 x 128 = 128

네트워크 주소: 192.168.1.128
브로드캐스트:  192.168.1.255 (128 + 128 - 1)
호스트 범위:   192.168.1.129 ~ 192.168.1.254
할당 가능:     2^7 - 2 = 126개
```

```
1단계: 호스트 비트 = 32 - 26 = 6비트
2단계: 블록 크기 = 2^6 = 64
3단계: 100 / 64 = 1...36 -> 1 x 64 = 64

네트워크 주소: 10.10.10.64
브로드캐스트:  10.10.10.127 (64 + 64 - 1)

```

### 예제 3: 172.16.5.200/28

```
1단계: 호스트 비트 = 32 - 28 = 4비트
2단계: 블록 크기 = 2^4 = 16
3단계: 200 / 16 = 12...8 -> 12 x 16 = 192

네트워크 주소: 172.16.5.192
브로드캐스트:  172.16.5.207 (192 + 16 - 1)
```

# 3장. 네트워크 개념

## 3.1 패킷 교환 방식 (가상회선 vs 데이터그램)

### 비교표

| 구분 | 가상회선 (Virtual Circuit) | 데이터그램 (Datagram) |
|------|---------------------------|----------------------|
| **경로 설정** | 전송 전에 경로 설정 (연결 설정) | 각 패킷이 독립적으로 경로 선택 |
| **패킷 순서** | 순서대로 도착 보장 | 순서 보장 안 됨 |
| **연결 방식** | 연결형 (Connection-oriented) | 비연결형 (Connectionless) |
| **대표 기술** | ATM, X.25 | IP, UDP |
| **장점** | 안정적, 순서 보장 | 유연, 장애 시 우회 가능 |
| **단점** | 경로 장애 시 전체 영향 | 순서 재조립 필요 |

**암기법:**
- **가상회선** = "**가**짜 **회선**" = 먼저 경로를 **예약**해놓고 순서대로 전송 = **전화**와 유사
- **데이터그램** = "**독립**적" = 각자 알아서 감 = **우편**과 유사

**시험에 이렇게 나온다:**
> Q. 패킷 교환 방식 중 전송 전에 경로를 설정하고 모든 패킷이 같은 경로로 전송되는 방식은?
> A. **가상회선 방식**

## 3.2 NAT (Network Address Translation)

```
[핵심 정리]
- 기능: 사설 IP 주소 <-> 공인 IP 주소 변환
- 목적: IP 주소 부족 문제 해결, 보안 강화
- 위치: 라우터 또는 방화벽에서 수행
```

```
내부 네트워크          NAT 라우터           외부 네트워크(인터넷)
192.168.1.10  ─┐
192.168.1.20  ─┼─> [NAT 변환] ─> 203.230.7.1 ─> 인터넷
192.168.1.30  ─┘   사설->공인
```

**사설 IP 주소 대역:**

| 클래스 | 사설 IP 범위 |
|--------|-------------|
| A | 10.0.0.0 ~ 10.255.255.255 |
| B | 172.16.0.0 ~ 172.31.255.255 |
| C | 192.168.0.0 ~ 192.168.255.255 |

**암기법:** "NAT = **N**etwork **A**ddress **T**ranslation = 주소 **번역** = 사설 <-> 공인"

## 3.3 VPN (Virtual Private Network)

```
[핵심 정리]
- 정의: 공중 네트워크(인터넷)를 통해 가상의 사설 네트워크를 구성
```

```
[핵심 정리]
```

# 4장. 보안 공격 유형 총정리 [최빈출]

## 4.1 네트워크 공격

### 세션 하이재킹 (Session Hijacking)

```
[핵심 정리]
- 정의: 정상적으로 인증된 세션을 탈취하여 불법 사용
- 방법: TCP 시퀀스 번호를 조작하여 세션을 가로챔
- 결과: 공격자가 정상 사용자인 척 시스템 접근
```

```
정상 사용자 <---세션---> 서버
                ↑
            공격자가 시퀀스 번호를 알아내서
            세션을 탈취 (하이재킹)
```

**암기법:** "하이재킹 = **비행기 납치** = **세션 납치** = **시퀀스 번호** 조작"

### SYN Flooding

```
[핵심 정리]
- TCP 3-Way Handshake를 악용한 DoS 공격
- 방법: SYN 패킷만 대량 전송, ACK는 보내지 않음
- 결과: 서버의 대기 큐(Backlog Queue)가 가득 차서 서비스 불능
- 유형: DoS (Denial of Service) 공격
```

```
공격자 --> SYN --> 서버 (SYN_RECEIVED 상태로 대기)
공격자 --> SYN --> 서버 (대기 큐에 쌓임)
공격자 --> SYN --> 서버 (대기 큐에 쌓임)
       ... ACK를 절대 안 보냄!
       --> 서버 대기 큐 가득 참 --> 서비스 거부!
```

**정상 TCP 3-Way Handshake와 비교:**
```
정상: SYN -> SYN+ACK -> ACK (연결 완료)
공격: SYN -> SYN+ACK -> ??? (ACK 안 보냄 = Half-Open)
```

**암기법:** "SYN Flooding = **SYN만** 보내고 **ACK 안 보냄** = 서버 **대기 큐 포화**"

### 스머프 공격 (Smurf Attack)

```
[핵심 정리]
- ICMP Echo Reply를 이용한 DDoS 공격
- 방법: 출발지 IP를 피해자 IP로 위조(스푸핑)하여
       브로드캐스트 주소로 ICMP Echo Request 전송
- 결과: 네트워크 내 모든 호스트가 피해자에게 Echo Reply를 보냄 -> 대량 트래픽
- 핵심: IP 스푸핑 + ICMP + 브로드캐스트
```

```
공격자 (출발지를 피해자 IP로 위조)
   │
   ├──> ICMP Echo Request ──> 브로드캐스트 주소
   │                            │
   │                     ┌──────┼──────┐
   │                     ▼      ▼      ▼
   │                   호스트1 호스트2 호스트3 ...
   │                     │      │      │
   │                     └──────┼──────┘
   │                            ▼
   │                     피해자 PC에 ICMP Echo Reply 폭주!
```

**암기법:** "스머프 = **ICMP** + **브로드캐스트** + **IP 위조** = Echo Reply 폭주"

### ICMP Flooding (Ping Flooding / Ping of Death)

```
[핵심 정리]
- ping 명령을 연속으로 대량 전송하는 공격
- ICMP Echo Request를 지속적으로 보내 네트워크 대역폭 소진
- Ping of Death: 규격 이상의 큰 ICMP 패킷 전송
```

## 4.2 지능형 공격: APT (Advanced Persistent Threat)

```
[핵심 정리]
- 정의: 지능형 지속 위협
- 특징: 특정 대상을 목표로 장기간에 걸쳐 지속적으로 공격
- 4단계: 침투 -> 탐색 -> 수집 -> 유출
```

### APT 4단계

| 단계 | 이름 | 설명 |
|------|------|------|
| 1단계 | **침투** (Infiltration) | 사회공학, 스피어피싱 등으로 내부 침투 |
| 2단계 | **탐색** (Search) | 내부 네트워크 구조, 취약점 탐색 |
| 3단계 | **수집** (Collection) | 중요 데이터 수집, 추가 권한 확보 |
| 4단계 | **유출** (Exfiltration) | 수집한 데이터를 외부로 유출 |

**암기법:** "APT = **침**투 -> **탐**색 -> **수**집 -> **유**출 = **침탐수유**"

# 5장. 악성코드 분류 [빈출]

## 웜 vs 트로이목마 vs 바이러스 비교표

| 구분 | 바이러스 (Virus) | 웜 (Worm) | 트로이 목마 (Trojan Horse) |
|------|-----------------|-----------|--------------------------|
| **자기 복제** | O (가능) | O (가능) | **X (불가)** |
| **감염 대상** | **필요** (숙주 파일) | 불필요 (**독자 실행**) | 불필요 |
| **전파 방식** | 파일 감염을 통해 | **네트워크**를 통해 자동 전파 | 사용자가 직접 실행 |
| **실행 조건** | 감염된 파일 실행 시 | **독자적으로** 실행 | 정상 프로그램으로 **위장** |
| **핵심 키워드** | 숙주 필요, 자기 복제 | **자가 복제, 독자 실행** | **정상 위장, 복제 불가** |

**시험 최빈출 구분 포인트:**
```
1. 자기 복제가 되는가?
   - YES -> 바이러스 또는 웜
   - NO  -> 트로이 목마

2. 숙주 파일이 필요한가?
   - YES -> 바이러스
   - NO  -> 웜 (독자 실행)
```

**암기법:**
- **바이러스** = "감기 **바이러스**" = **숙주**(사람 몸)가 필요 = 자기 복제 O

# 6장. 암호화 알고리즘 [빈출]

## 6.1 대칭키 vs 비대칭키

| 구분 | 대칭키 (비밀키) | 비대칭키 (공개키) |
|------|-----------------|-------------------|
| **키 사용** | 암호화/복호화에 **같은 키** 사용 | **공개키**로 암호화, **개인키**로 복호화 |
| **키 개수** | n명일 때: **n(n-1)/2개** | n명일 때: **2n개** |
| **속도** | **빠름** | 느림 |
| **키 분배** | 키 분배 문제 있음 | 키 분배 문제 없음 |
| **대표 알고리즘** | DES, AES, ARIA, SEED | RSA, ECC |
| **용도** | 대량 데이터 암호화 | 키 교환, 전자서명 |

**암기법:**
- **대칭키** = **대**칭 = **같**은 키 = **DAS**(DES, AES, SEED, ARIA)
- **비대칭키** = **비**대칭 = **다른** 키(공개+개인) = **RSA, ECC**

---

## 6.2 주요 알고리즘 특징

### 대칭키 알고리즘

| 알고리즘 | 핵심 특징 |
|----------|-----------|
| **DES** | 64비트 블록, 56비트 키, IBM 개발, 현재는 취약하여 AES로 대체 |
| **AES** | **NIST 표준**, DES 대체, **128비트 블록**, 키 길이 128/192/256비트, Rijndael 알고리즘 |
| **ARIA** | 한국 개발 (국가정보원+학계), 128비트 블록, 키 128/192/256비트 |
| **SEED** | 한국 개발 (KISA), 128비트 블록, 128비트 키 |
| **3DES** | DES를 3번 반복 적용, DES보다 안전하지만 느림 |

### 비대칭키 알고리즘

| 알고리즘 | 핵심 특징 |
|----------|-----------|
| **RSA** | 소인수분해의 어려움 기반, 가장 대표적 공개키 알고리즘 |
| **ECC** | 타원곡선 암호, RSA보다 짧은 키로 동일 보안 수준, 모바일 환경 적합 |

## 6.3 AES 상세 (시험 단골!)

```
[AES 핵심 정리]
- 풀네임: Advanced Encryption Standard
- 제정: NIST (미국 국립표준기술연구소)
- 목적: DES를 대체하기 위해 개발
- 블록 크기: 128비트 (고정)
- 키 길이: 128 / 192 / 256비트 (선택 가능)
- 라운드 수: 10 / 12 / 14라운드 (키 길이에 따라)
- 원래 이름: Rijndael (레인달)
- 방식: 대칭키 블록 암호
```

| 키 길이 | 라운드 수 |
|---------|-----------|
| 128비트 | 10라운드 |
| 192비트 | 12라운드 |
| 256비트 | 14라운드 |

**암기법:** "AES = **A**dvanced = DES **대체** = **NIST** 표준 = 128비트 블록 = 128/192/256"

# 7장. 접근 통제 모델 [최빈출! 거의 매회 출제]

## MAC vs DAC vs RBAC 완벽 비교

| 구분 | MAC | DAC | RBAC |
|------|-----|-----|------|
| **풀네임** | Mandatory Access Control | Discretionary Access Control | Role-Based Access Control |
| **한글** | **강제적** 접근 통제 | **임의적** 접근 통제 | **역할 기반** 접근 통제 |
| **권한 부여** | **시스템(중앙 관리자)**이 부여 | **소유자(자원 소유자)**가 부여 | **역할(직책)**에 따라 부여 |
| **보안 등급** | 보안 등급(레이블) 기반 | 신분(Identity) 기반 | 역할(Role) 기반 |
| **유연성** | 낮음 (엄격) | 높음 (유연) | 중간 |
| **보안 수준** | **가장 높음** | 상대적으로 낮음 | 중간 |
| **적용 환경** | 군사/정부 기관 | 일반 상용 시스템 | 기업 조직 |
| **대표 모델** | Bell-LaPadula, Biba | ACL (Access Control List) | - |
| **특징** | 관리자만 권한 변경 가능 | 소유자가 다른 사용자에게 권한 부여 가능 | 사용자를 역할에 할당 |

### 핵심 구분 포인트

```
누가 권한을 부여하는가?
  - 시스템/중앙 관리자 -> MAC (강제적)
  - 자원의 소유자      -> DAC (임의적)
  - 역할/직책 기반     -> RBAC (역할 기반)
```

### 암기법

```
MAC  = Mandatory  = 강제적 = 군대! = 상관(시스템)이 시킴 = 보안 등급
DAC  = Discretionary = 임의적 = 자유! = 소유자가 알아서 = 신분 기반
RBAC = Role-Based = 역할! = 회사! = 부장/과장/사원 역할에 따라 = 역할 기반
```

**초간단 암기:**
- **MAC** = **M**ilitary(군사) = **강제** = **중앙** 관리
- **DAC** = **D**iscretion(재량) = **임의** = **소유자** 재량
- **RBAC** = **R**ole(역할) = **역할/직책** 기반

**시험에 이렇게 나온다:**
> Q. 시스템이 주체와 객체의 보안 등급을 비교하여 접근 권한을 부여하는 강제적 접근 통제 방식은?
> A. **MAC (Mandatory Access Control)**

> Q. 데이터 소유자가 자신의 판단에 따라 다른 사용자에게 접근 권한을 부여하는 방식은?
> A. **DAC (Discretionary Access Control)**

> Q. 조직 내 역할(Role)에 기반하여 접근 권한을 부여하는 방식은?
> A. **RBAC (Role-Based Access Control)**

## 8.1 OAuth

```
[핵심 정리]
- 정의: 개방형 표준 인증 프로토콜
- 핵심: 제3자(Third-party) 인증, 비밀번호를 직접 제공하지 않음
```

```
2. 구글 인증 페이지로 이동 (인증 요청)
3. 사용자가 구글에서 직접 로그인
4. 구글이 서비스A에 Access Token 발급
5. 서비스A가 Access Token으로 구글 API 접근
```

**암기법:** "OAuth = **O**pen **Auth**orization = **개방형** 인증 = **Access Token** = 제3자 인증"

## 8.2 OTP (One-Time Password)

```
[핵심 정리]
- 정의: 일회용 비밀번호 (매번 새로운 비밀번호 사용)
- 생성 방식:
  1. 시간 동기화 방식 (TOTP): 시간 기반으로 비밀번호 생성
  2. 카운터 기반 방식 (HOTP): 사용 횟수 기반으로 생성

```

**암기법:** "OTP = **O**ne **T**ime **P**assword = **일회용** = 한 번 쓰면 끝"

## 8.3 해시 (Hash)

```
[핵심 정리]
- 정의: 임의 길이의 데이터를 고정 길이의 값으로 변환하는 함수
- 특징:
  - 단방향 함수 (역변환 불가능)
```

| SHA-256 | 256비트 | 현재 표준 |
| SHA-512 | 512비트 | 높은 보안 |

**암기법:** "해시 = **고정 길이** + **단방향** + **무결성**"

# 9장. 오류 검출/정정 [빈출]

## 오류 제어 방식 개요

```
오류 제어
├── 오류 검출 (Detection)
│   ├── Parity Check (패리티 검사)
│   └── CRC (순환 중복 검사)
├── 오류 정정 (Correction)
│   └── Hamming Code (해밍 코드)
└── 오류 제어 방식
    ├── FEC (Forward Error Correction) - 전진 오류 수정
    └── BEC (Backward Error Correction) - 후진 오류 수정
```

## 9.1 CRC (Cyclic Redundancy Check)

```
[핵심 정리]
- 정의: 순환 중복 검사
- 방식: 다항식(Polynomial) 기반으로 검사 값 생성
- 특징: 오류 검출만 가능 (정정은 불가)
- 신뢰도: 패리티보다 높은 오류 검출 능력
- 적용: 네트워크 프레임, 저장 장치
```

**암기법:** "CRC = **C**yclic(순환) + **R**edundancy(중복) + **C**heck(검사) = **다항식** 기반 = 검출만!"

## 9.2 Hamming Code (해밍 코드)

```
[핵심 정리]
- 정의: 오류를 검출하고 수정(정정)까지 할 수 있는 코드
- 특징: 1비트 오류 수정 가능, 2비트 오류 검출 가능
- 방식: 데이터 비트 사이에 검사 비트(패리티 비트)를 삽입
- 핵심: CRC는 검출만, Hamming은 정정까지!
```

**암기법:** "Hamming = **해밍** = **해결**(수정)까지 하는 밍(놈) = 오류 **검출 + 정정**"

## 9.3 Parity Check (패리티 검사)

```
[핵심 정리]
- 정의: 데이터에 패리티 비트(Parity Bit)를 추가하여 오류 검출
- 종류:
  - 짝수 패리티: 1의 개수가 짝수가 되도록
  - 홀수 패리티: 1의 개수가 홀수가 되도록
- 한계: 1비트 오류만 검출, 짝수 개 오류는 검출 불가, 정정 불가
```

## 9.4 FEC vs BEC

| 구분 | FEC (전진 오류 수정) | BEC (후진 오류 수정) |
|------|---------------------|---------------------|
| **풀네임** | Forward Error Correction | Backward Error Correction |
| **방식** | **수신측이 스스로 오류를 수정** | **재전송을 요청**하여 오류 해결 |
| **재전송** | 필요 없음 | **재전송 필요** |
| **대표 방식** | Hamming Code, 상승 부호 | ARQ (Stop-and-Wait, Go-Back-N, Selective Repeat) |
| **장점** | 실시간 처리 가능 | 구현 간단 |
| **단점** | 복잡, 추가 비트 필요 | 재전송으로 인한 지연 |

**암기법:**
- **FEC** = **F**orward(전진) = **앞으로** 가기만 함 = 재전송 없이 **스스로 수정**
- **BEC** = **B**ackward(후진) = **뒤로** 돌아감 = **재전송 요청**

### ARQ (Automatic Repeat Request) - BEC의 대표 기법

| 방식 | 설명 |
|------|------|
| **Stop-and-Wait** | 한 프레임 보내고 ACK 기다림 (가장 단순) |
| **Go-Back-N** | 오류 발생 프레임부터 모두 재전송 |
| **Selective Repeat** | 오류 발생 프레임만 선택적 재전송 |

|------|--------|-----|---------|
| 오류 검출 | O (1비트) | O (강력) | O |
| **오류 정정** | **X** | **X** | **O (1비트)** |
| 방식 | 패리티 비트 | 다항식 | 검사 비트 삽입 |
| 신뢰도 | 낮음 | 높음 | 높음 |

# 10장. 기타 보안 개념

## 템퍼프루핑 (Tamper Proofing)

```
[핵심 정리]
- 정의: 소프트웨어가 불법적으로 변경(Tamper)되었을 때
       정상적으로 수행되지 못하도록 하는 기법
- Tamper = 변조, Proofing = 방지
- 자가 무결성 검증: 소프트웨어가 스스로 변조 여부를 확인
- 변조 감지 시: 실행 중단, 기능 제한, 데이터 삭제 등
```

**암기법:** "Tamper Proofing = **변조 방지** = 불법 변경 시 **정상 수행 불가**"

> - ( 2 ): MAC 주소를 이용하여 IP 주소를 알아내는 프로토콜

**정답: (1) ARP  (2) RARP**

## 문제 4. [서브넷 계산]
> IP 주소가 192.168.10.150이고 서브넷 마스크가 255.255.255.224(/27)일 때,
> (1) 네트워크 주소
> (2) 브로드캐스트 주소
> (3) 할당 가능한 호스트 수를 구하시오.

**풀이:**
```
블록 크기 = 256 - 224 = 32
150 / 32 = 4...22 -> 4 x 32 = 128

(1) 네트워크 주소: 192.168.10.128
(2) 브로드캐스트: 192.168.10.159 (128 + 32 - 1)
(3) 호스트 수: 2^5 - 2 = 30개
```
