## 1장. 관계형 데이터베이스 핵심 용어 ★★★

> **출제 빈도: 거의 매회 | 난이도: 하 | 체감 점수: 공짜**

### 1.1 릴레이션 구조 한눈에 보기

```
                    ← ─ ─ ─ Degree(차수) = 4 ─ ─ ─ →

              ┌──────────┬───────┬──────┬────────────┐
              │  학번    │ 이름  │ 학과 │ 전화번호    │  ← Attribute(속성) = Column
              ├──────────┼───────┼──────┼────────────┤
   Tuple ──→  │  2001    │ 김철수│ 컴공 │ 010-1234   │  ← Tuple(튜플) = Row
   (행)       │  2002    │ 이영희│ 전자 │ 010-5678   │
              │  2003    │ 박민수│ 컴공 │ 010-9012   │
              │  2004    │ 정다은│ 수학 │ 010-3456   │
              └──────────┴───────┴──────┴────────────┘
                    ↑
                Cardinality(카디널리티) = 4 (튜플의 수)

   ■ Relation(릴레이션) = 테이블 전체
   ■ Schema(스키마)     = 학생(학번, 이름, 학과, 전화번호) ← 구조 정의
   ■ Instance(인스턴스)  = 특정 시점의 실제 데이터 (위 4개 행)
   ■ Domain(도메인)      = 속성이 가질 수 있는 값의 범위
     - 학과 도메인: {컴공, 전자, 수학, 물리, 화학, ...}
     - 학번 도메인: {2001~9999}
```

### 1.2 핵심 용어 완벽 정리

#### Degree(차수) - 속성(컬럼)의 수

```
★ 암기법: "디그리는 세로줄 수" (Degree = 열/컬럼 수)
★ 또는: "디(D)그리 = (아래로)Down = 세로 = 속성 수"
```

| 릴레이션 | 속성 | Degree |
|---------|------|:------:|
| 학생(학번, 이름, 학과) | 학번, 이름, 학과 | **3** |
| 성적(학번, 과목, 점수, 등급) | 학번, 과목, 점수, 등급 | **4** |
| 사원(사번, 이름, 부서, 직급, 급여) | 사번, 이름, 부서, 직급, 급여 | **5** |

#### Cardinality(카디널리티) - 튜플(행)의 수

```
★ 암기법: "카디널리티는 가로줄 수" (Cardinality = 행/튜플 수)
★ 또는: "Car(자동차)는 가로로 달린다 = 행의 수"
```

| 릴레이션 | 데이터 행 수 | Cardinality |
|---------|:----------:|:-----------:|
| 학생 3명 등록 | 3행 | **3** |
| 과목 10개 등록 | 10행 | **10** |
| 빈 테이블 | 0행 | **0** |

### 1.3 혼동하기 쉬운 용어 비교

#### Degree vs Cardinality (매회 출제!)

```
┌─────────────────────────────────────────────┐
│         Degree = 속성 수 = 열(Column) 수    │
│    Cardinality = 튜플 수 = 행(Row) 수       │
│                                              │
│  ★ 기출 함정: "Degree는 행의 수이다" → (X)  │
│  ★ 기출 함정: "Cardinality는 열의 수" → (X) │
└─────────────────────────────────────────────┘
```

#### 같은 의미, 다른 용어 (동의어 정리)

| 관계형 DB 용어 | 파일 시스템 용어 | 일반 용어 |
|:---:|:---:|:---:|
| 릴레이션(Relation) | 파일(File) | 테이블(Table) |
| 튜플(Tuple) | 레코드(Record) | 행(Row) |
| 속성(Attribute) | 필드(Field) | 열(Column) |
| 도메인(Domain) | - | 값의 범위 |

#### Tuple vs Record vs Row

```
  세 용어 모두 같은 의미! (행 = 가로 한 줄의 데이터)
  - Tuple    → 관계형 DB 이론 용어
  - Record   → 파일 시스템 용어
  - Row      → 실무/SQL 용어

  ★ 시험에서는 "튜플"로 가장 많이 출제
```

#### Attribute vs Domain

```
  - Attribute(속성) = 열 자체 (이름, 학과, 나이 등 컬럼명)
  - Domain(도메인)  = 그 열에 들어갈 수 있는 값의 범위

  예시:
    Attribute: 성별       → Domain: {남, 여}
    Attribute: 학년       → Domain: {1, 2, 3, 4}
    Attribute: 학점       → Domain: {A+, A, B+, B, C+, C, D+, D, F}
```

#### Schema vs Instance

```
  - Schema(스키마)     = 릴레이션의 "구조 정의" (변하지 않음)
    예: 학생(학번 INT, 이름 VARCHAR, 학과 VARCHAR)

  - Instance(인스턴스) = 특정 시점의 "실제 데이터" (수시로 변함)
    예: (2001, '김철수', '컴공'), (2002, '이영희', '전자'), ...
```

### 🔥 이것만 외워! 핵심 암기표

| 용어 | 의미 | 동의어 | 기출횟수 | 암기법 |
|:---:|:---:|:---:|:---:|:---:|
| **Degree(차수)** | 속성(컬럼)의 수 | - | 8회 | D=Down=세로=열 수 |
| **Cardinality(카디널리티)** | 튜플(행)의 수 | - | 8회 | Car=가로=행 수 |
| **Tuple(튜플)** | 행(Row) | Record, Row | 7회 | 튜플=가로 한 줄 |
| **Attribute(속성)** | 열(Column) | Field, Column | 7회 | 속성=세로 한 줄 |
| **Domain(도메인)** | 속성의 값 범위 | - | 5회 | 도메인=값의 영역 |
| **Schema(스키마)** | 릴레이션 구조 정의 | - | 4회 | 스키마=설계도 |
| **Instance(인스턴스)** | 특정 시점 데이터 | - | 3회 | 인스턴스=실제 데이터 |
| **Relation(릴레이션)** | 테이블 | File, Table | 3회 | 릴레이션=표 |

### 🎯 기출 확인 문제

```
[문제] 다음 릴레이션에서 Degree와 Cardinality를 구하시오.

  사원(사번, 이름, 부서, 직급)
  ┌──────┬───────┬──────┬──────┐
  │ 1001 │ 김철수 │ 영업  │ 대리 │
  │ 1002 │ 이영희 │ 개발  │ 과장 │
  │ 1003 │ 박민수 │ 인사  │ 사원 │
  └──────┴───────┴──────┴──────┘

[정답] Degree = 4 (속성 수: 사번, 이름, 부서, 직급)
       Cardinality = 3 (튜플 수: 3개의 행)
```

## 2장. 키(Key) 완벽 정리 ★★★

> **출제 빈도: 거의 매회 | 난이도: 하 | 체감 점수: 공짜**

### 2.1 키의 종류와 관계도

```
    ┌──────────────────────────────────────────┐
    │            슈퍼키 (Super Key)            │
    │   유일성 O, 최소성 X (속성 조합 가능)    │
    │                                          │
    │    ┌──────────────────────────────┐      │
    │    │      후보키 (Candidate Key)  │      │
    │    │   유일성 O, 최소성 O         │      │
    │    │                              │      │
    │    │  ┌────────┐  ┌────────────┐ │      │
    │    │  │ 기본키  │  │  대체키    │ │      │
    │    │  │(Primary │  │(Alternate  │ │      │
    │    │  │  Key)   │  │  Key)      │ │      │
    │    │  │ 선택된  │  │ 선택 안된  │ │      │
    │    │  │ 후보키  │  │ 나머지     │ │      │
    │    │  └────────┘  └────────────┘ │      │
    │    └──────────────────────────────┘      │
    └──────────────────────────────────────────┘

    ┌──────────────────────────────────────────┐
    │         외래키 (Foreign Key)              │
    │   다른 릴레이션의 기본키를 참조           │
    └──────────────────────────────────────────┘
```

### 2.2 각 키 상세 설명

#### 슈퍼키 (Super Key)

```
정의: 튜플을 유일하게 식별할 수 있는 속성 또는 속성 집합
특징: 유일성 O, 최소성 X (불필요한 속성 포함 가능)

예시: 학생(학번, 주민번호, 이름, 학과)
  - {학번}              → 슈퍼키 O (유일 식별 가능)
  - {주민번호}          → 슈퍼키 O
  - {학번, 이름}        → 슈퍼키 O (이름 불필요하지만 OK)
  - {학번, 주민번호, 이름} → 슈퍼키 O (과잉이지만 OK)
  - {이름}              → 슈퍼키 X (동명이인 가능 → 유일 식별 불가)
  - {학과}              → 슈퍼키 X (같은 학과 여러 명)

★ 핵심: 유일하게 구별만 되면 OK. 속성을 마구 추가해도 됨!
```

#### 후보키 (Candidate Key)

```
정의: 유일성 + 최소성을 모두 만족하는 속성 집합
특징: 슈퍼키 중 최소한의 속성으로 구성된 것

예시: 학생(학번, 주민번호, 이름, 학과)
  - {학번}       → 후보키 O (유일성 O, 최소성 O)
  - {주민번호}   → 후보키 O (유일성 O, 최소성 O)
  - {학번, 이름} → 후보키 X (학번만으로 식별 가능 → 최소성 X)

★ 핵심: "더 줄일 수 없는" 슈퍼키 = 후보키
```

#### 기본키 (Primary Key)

```
정의: 후보키 중 대표로 선택된 하나의 키
특징:
  (1) NULL 값 불허 (반드시 값 존재)
  (2) 중복 불허 (모든 행이 다른 값)
  (3) 변경 최소화 (가급적 변하지 않는 속성)

예시: 학생(학번, 주민번호, 이름, 학과)
  - 후보키: {학번}, {주민번호}
  - 기본키: {학번} 선택 (더 간결하고 변경 가능성 낮음)
  - 대체키: {주민번호} (선택되지 않은 후보키)
```

#### 대체키 (Alternate Key)

```
정의: 후보키 중 기본키로 선택되지 않은 나머지 키
공식: 대체키 = 후보키 - 기본키

예시:
  후보키: {학번}, {주민번호}
  기본키: {학번}
  대체키: {주민번호}

★ 암기: "대체"할 수 있는 키 = 기본키 대신 쓸 수 있었던 키
```

#### 외래키 (Foreign Key)

```
정의: 다른 릴레이션의 기본키를 참조하는 속성
특징:
  (1) 참조되는 릴레이션의 기본키 값이거나 NULL이어야 함
  (2) 릴레이션 간 관계를 표현

예시:
  학생(학번(PK), 이름, 학과코드(FK))
  학과(학과코드(PK), 학과명, 위치)

  학생 테이블의 '학과코드'는 학과 테이블의 '학과코드(PK)'를 참조
  → 학생.학과코드 = 외래키(FK)

★ 주의: 외래키는 자기 자신 테이블의 PK를 참조할 수도 있음 (셀프 참조)
★ 주의: 외래키는 NULL 값 가능! (기본키는 NULL 불가)
```

### 2.3 유일성과 최소성

```
┌──────────┬────────┬────────┐
│   키     │ 유일성 │ 최소성 │
├──────────┼────────┼────────┤
│ 슈퍼키   │   O    │   X    │  ← 유일하기만 하면 됨
│ 후보키   │   O    │   O    │  ← 유일 + 최소
│ 기본키   │   O    │   O    │  ← 후보키 중 대표 1개
│ 대체키   │   O    │   O    │  ← 후보키 중 나머지
│ 외래키   │   -    │   -    │  ← 다른 테이블 PK 참조
└──────────┴────────┴────────┘

★ 유일성: 각 튜플을 유일하게 식별 가능한가?
★ 최소성: 최소한의 속성으로 구성되었는가? (불필요 속성 없는가?)
```

### 🔥 키 관계 한눈에 보기

```
  슈퍼키 ⊃ 후보키 ⊃ { 기본키, 대체키 }

  "슈후기대" 로 외우자!
    슈(퍼키) → 후(보키) → 기(본키) / 대(체키)

  ★ 슈퍼키가 가장 큰 범위
  ★ 후보키는 슈퍼키의 부분집합
  ★ 기본키와 대체키는 후보키의 부분집합
  ★ 외래키는 별도 (다른 테이블 PK 참조)
```

### 🎯 기출 확인 문제

```
[문제1] 릴레이션 R(A, B, C, D)에서
        후보키가 {A, B}와 {C}일 때,
        기본키를 {C}로 선택하면 대체키는?

[정답] 대체키 = {A, B}
       (후보키 중 기본키로 선택되지 않은 키)

[문제2] 다음 중 외래키에 대한 설명으로 옳지 않은 것은?
  ① 다른 릴레이션의 기본키를 참조한다
  ② NULL 값을 가질 수 있다
  ③ 반드시 유일한 값이어야 한다      ← 정답 (X)
  ④ 참조 무결성 제약조건과 관련된다

[해설] 외래키는 중복 가능! 여러 학생이 같은 학과코드를 가질 수 있음
```

## 3장. 무결성 제약조건 ★★★

> **출제 빈도: 거의 매회 | 난이도: 하 | 체감 점수: 공짜**

### 3.1 개체 무결성 (Entity Integrity)

```
정의: 기본키(PK)는 NULL이 될 수 없고, 중복될 수 없다

규칙:
  (1) 기본키에 NULL 값 불허
  (2) 기본키에 중복 값 불허

예시: 학생(학번(PK), 이름, 학과)
  ┌──────┬───────┬──────┐
  │ 학번 │ 이름  │ 학과 │
  ├──────┼───────┼──────┤
  │ 2001 │ 김철수│ 컴공 │  ← OK
  │ NULL │ 이영희│ 전자 │  ← 위반! (PK에 NULL)
  │ 2001 │ 박민수│ 수학 │  ← 위반! (PK 중복)
  └──────┴───────┴──────┘

★ 암기: "개체 = 기본키 = NULL X, 중복 X"
★ "개.기.널" (개체-기본키-NULL불가)
```

### 3.2 참조 무결성 (Referential Integrity)

```
정의: 외래키(FK) 값은 참조하는 릴레이션의 기본키 값이거나 NULL이어야 한다

규칙:
  (1) FK 값은 참조 테이블의 PK에 존재하는 값이어야 함
  (2) 또는 NULL이어야 함 (FK는 NULL 허용!)

예시:
  학과(학과코드(PK), 학과명)         학생(학번(PK), 이름, 학과코드(FK))
  ┌──────────┬──────┐              ┌──────┬───────┬──────────┐
  │ 학과코드 │학과명│              │ 학번 │ 이름  │ 학과코드 │
  ├──────────┼──────┤              ├──────┼───────┼──────────┤
  │    C1    │ 컴공 │              │ 2001 │ 김철수│    C1    │ ← OK (C1 존재)
  │    E1    │ 전자 │              │ 2002 │ 이영희│    E1    │ ← OK (E1 존재)
  │    M1    │ 수학 │              │ 2003 │ 박민수│   NULL   │ ← OK (NULL 허용)
  └──────────┴──────┘              │ 2004 │ 정다은│    P1    │ ← 위반! (P1 없음)
                                   └──────┴───────┴──────────┘

★ 암기: "참조 = 외래키 = 참조 PK 값 or NULL"
★ "참.외.참" (참조-외래키-참조값존재)
```

### 3.3 도메인 무결성 (Domain Integrity)

```
정의: 속성 값이 정의된 도메인(값의 범위)에 속해야 한다

규칙: 각 속성의 값은 해당 도메인 안의 값이어야 함

예시:
  성별 도메인: {남, 여}
  학년 도메인: {1, 2, 3, 4}
  나이 도메인: 1~150 사이 정수

  학생(학번, 이름, 성별, 학년)
  ┌──────┬───────┬──────┬──────┐
  │ 학번 │ 이름  │ 성별 │ 학년 │
  ├──────┼───────┼──────┼──────┤
  │ 2001 │ 김철수│  남  │  2   │ ← OK
  │ 2002 │ 이영희│  양  │  3   │ ← 위반! (성별 도메인에 '양' 없음)
  │ 2003 │ 박민수│  남  │  7   │ ← 위반! (학년 도메인 1~4인데 7)
  └──────┴───────┴──────┴──────┘

★ 암기: "도메인 = 값의 범위 안에 있어야"
```

### 🔥 무결성 3종 비교표

| 구분 | 개체 무결성 | 참조 무결성 | 도메인 무결성 |
|:---:|:---:|:---:|:---:|
| **대상** | 기본키(PK) | 외래키(FK) | 속성 값 |
| **조건** | NULL 불가, 중복 불가 | 참조 PK값 or NULL | 도메인 범위 내 |
| **암기** | 개-기-널 | 참-외-참 | 도-값-범 |
| **출제빈도** | 초빈출 | 초빈출 | 빈출 |

```
★ 초고속 암기법: "개기참외도값"
  개(체)기(본키) / 참(조)외(래키) / 도(메인)값(범위)
```

### 🔥 무결성 위반 판단 문제 풀이법

```
Step 1: 어떤 키에 대한 문제인지 파악
  → PK 관련이면 → 개체 무결성 체크
  → FK 관련이면 → 참조 무결성 체크
  → 값 범위면   → 도메인 무결성 체크

Step 2: 위반 조건 확인
  - PK에 NULL? → 개체 무결성 위반
  - PK에 중복? → 개체 무결성 위반
  - FK 값이 참조 PK에 없고 NULL도 아님? → 참조 무결성 위반
  - 속성 값이 도메인 밖? → 도메인 무결성 위반
```

### 🎯 기출 확인 문제

```
[문제] 다음 삽입 연산 중 무결성 제약조건을 위반하는 것과
      위반하는 무결성의 이름을 쓰시오.

  학생(학번(PK), 이름, 학과코드(FK))
  학과(학과코드(PK), 학과명)  -- 학과코드: {C1, E1, M1}

  (가) INSERT INTO 학생 VALUES(NULL, '홍길동', 'C1');
  (나) INSERT INTO 학생 VALUES(3001, '임꺽정', 'P1');
  (다) INSERT INTO 학생 VALUES(3002, '장보고', NULL);

[정답]
  (가) 개체 무결성 위반 (PK인 학번에 NULL)
  (나) 참조 무결성 위반 (FK인 학과코드 'P1'이 학과 테이블에 없음)
  (다) 위반 없음 (FK는 NULL 허용)
```

## 4장. 정규화 & 반정규화 ★★☆

> **출제 빈도: 5/9회 | 난이도: 중 | 핵심: 위반 조건 판별**

### 4.1 정규화란?

```
정의: 데이터 중복을 최소화하고, 이상(Anomaly)을 제거하기 위해
      릴레이션을 분해하는 과정

이상(Anomaly)의 종류:
  (1) 삽입 이상: 불필요한 데이터까지 삽입해야 하는 문제
  (2) 삭제 이상: 필요한 데이터까지 같이 삭제되는 문제
  (3) 갱신 이상: 일부만 변경하면 데이터 불일치 발생

★ 암기: "이상을 없애려고 정규화한다"
```

### 4.2 정규화 단계 (1NF → 2NF → 3NF → BCNF)

```
★ 암기 순서: "도부이결다조" (두부이걸다줘)
  제1정규형: 도메인이 원자값
  제2정규형: 부분 함수 종속 제거
  제3정규형: 이행 함수 종속 제거
  BCNF:     결정자가 모두 후보키
  제4정규형: 다치 종속 제거
  제5정규형: 조인 종속 제거

※ 시험 범위: 주로 1NF ~ BCNF까지 출제
```

#### 제1정규형 (1NF)

```
조건: 모든 속성의 도메인이 원자값(Atomic Value)으로만 구성

위반 예시 (1NF 아님):
  ┌──────┬───────┬─────────────────┐
  │ 학번 │ 이름  │     과목        │
  ├──────┼───────┼─────────────────┤
  │ 2001 │ 김철수│ {DB, OS, 자료}  │  ← 다중 값! (원자값 아님)
  │ 2002 │ 이영희│ {DB, 네트워크}  │
  └──────┴───────┴─────────────────┘

정규화 후 (1NF):
  ┌──────┬───────┬──────────┐
  │ 학번 │ 이름  │   과목   │
  ├──────┼───────┼──────────┤
  │ 2001 │ 김철수│   DB     │  ← 원자값!
  │ 2001 │ 김철수│   OS     │
  │ 2001 │ 김철수│   자료   │
  │ 2002 │ 이영희│   DB     │
  │ 2002 │ 이영희│ 네트워크 │
  └──────┴───────┴──────────┘

★ 핵심: "하나의 셀에 하나의 값만!" = 1NF
```

#### 제2정규형 (2NF)

```
조건: 1NF + 부분 함수 종속 제거 (완전 함수 종속)

부분 함수 종속이란?
  기본키가 복합키(2개 이상)일 때,
  기본키의 "일부분"에만 종속되는 속성이 있는 경우

위반 예시 (1NF이지만 2NF 아님):
  수강(학번, 과목코드, 성적, 과목명)
  PK = {학번, 과목코드}

  학번, 과목코드 → 성적     (완전 함수 종속 O)
  과목코드 → 과목명          (부분 함수 종속! 기본키 일부에 종속)

정규화 후 (2NF):
  수강(학번, 과목코드, 성적)      PK = {학번, 과목코드}
  과목(과목코드, 과목명)          PK = {과목코드}

★ 핵심: "복합키의 일부에 종속 → 분리!" = 2NF
★ 참고: 기본키가 단일 속성이면 자동으로 2NF 만족
```

#### 제3정규형 (3NF)

```
조건: 2NF + 이행 함수 종속 제거

이행 함수 종속이란?
  A → B, B → C 일 때, A → C가 성립하는 관계
  (A가 B를 결정하고, B가 C를 결정하면, A가 C도 결정)

위반 예시 (2NF이지만 3NF 아님):
  학생(학번, 이름, 학과코드, 학과명)
  PK = {학번}

  학번 → 학과코드     (직접 종속)
  학과코드 → 학과명   (직접 종속)
  학번 → 학과명       (이행 종속! A→B→C)

정규화 후 (3NF):
  학생(학번, 이름, 학과코드)   PK = {학번}
  학과(학과코드, 학과명)       PK = {학과코드}

★ 핵심: "A→B→C 이행 종속 → 분리!" = 3NF
★ 암기: "이행(이사행) 가면 분리"
```

#### BCNF (Boyce-Codd Normal Form)

```
조건: 3NF + 모든 결정자가 후보키

결정자란?
  함수 종속에서 다른 속성을 결정하는 속성 (화살표 왼쪽)
  A → B 에서 A가 결정자

위반 예시 (3NF이지만 BCNF 아님):
  수강(학번, 과목, 담당교수)
  PK = {학번, 과목}

  {학번, 과목} → 담당교수   (OK, PK가 결정자)
  담당교수 → 과목           (문제! 담당교수는 결정자인데 후보키 아님)

정규화 후 (BCNF):
  수강(학번, 담당교수)    PK = {학번, 담당교수}
  교수담당(담당교수, 과목) PK = {담당교수}

★ 핵심: "결정자인데 후보키 아닌 놈이 있으면 → BCNF 위반"
★ BCNF = 3NF를 강화한 버전
```

### 4.3 반정규화 (Denormalization / 비정규화)

```
정의: 정규화된 릴레이션을 성능 향상을 위해 의도적으로 통합/중복하는 과정

이유:
  - 정규화 → 테이블 분리 많음 → JOIN 많아짐 → 성능 저하
  - 읽기(조회) 성능이 중요한 경우 반정규화 적용

방법:
  (1) 테이블 병합 (분리된 테이블을 다시 합침)
  (2) 중복 속성 추가 (자주 조회하는 속성을 중복 저장)
  (3) 파생 속성 추가 (계산 결과를 미리 저장)

★ 핵심: "정규화 ↔ 반정규화 = 트레이드오프"
  정규화: 중복 최소화, 무결성 보장 (쓰기 유리)
  반정규화: 성능 향상, 중복 허용 (읽기 유리)
```

### 🔥 "몇 정규형인가?" 판단법

```
Step 1: 원자값인가?
  → 다중 값 있으면 → 비정규형 (1NF 미달)
  → 원자값이면 → 최소 1NF

Step 2: 부분 함수 종속 있는가? (복합키일 때)
  → 있으면 → 1NF (2NF 미달)
  → 없으면 → 최소 2NF

Step 3: 이행 함수 종속 있는가?
  → 있으면 → 2NF (3NF 미달)
  → 없으면 → 최소 3NF

Step 4: 결정자 중 후보키가 아닌 것이 있는가?
  → 있으면 → 3NF (BCNF 미달)
  → 없으면 → BCNF

★ 판단 순서: 원자값 → 부분종속 → 이행종속 → 결정자=후보키
```

### 🎯 기출 확인 문제

```
[문제] 다음 릴레이션은 제 몇 정규형인지 쓰시오.

  수강(학번, 과목코드, 과목명, 성적)
  PK = {학번, 과목코드}
  함수 종속: {학번, 과목코드} → 성적, 과목코드 → 과목명

[풀이]
  1) 원자값? → O → 1NF 만족
  2) 부분 함수 종속?
     과목코드 → 과목명 (기본키 일부에 종속) → 부분 종속 있음!
  3) 따라서 1NF (2NF 미달)

[정답] 제1정규형 (1NF)
```

## 5장. 스키마 3계층 ★★☆

> **출제 빈도: 4/9회 | 난이도: 하 | 비교 문제 출제**

### 5.1 외부/개념/내부 스키마

```
  ┌─────────────────────────────────────────┐
  │     사용자A    사용자B    사용자C       │
  │        │          │          │          │
  │     ┌──▼──┐   ┌──▼──┐   ┌──▼──┐       │
  │     │외부1│   │외부2│   │외부3│  외부 스키마 (여러 개)
  │     └──┬──┘   └──┬──┘   └──┬──┘       │
  │        └─────────┼─────────┘           │
  │     ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  논리적 독립성
  │              ┌───▼───┐                  │
  │              │ 개념  │  개념 스키마 (1개)│
  │              └───┬───┘                  │
  │     ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  물리적 독립성
  │              ┌───▼───┐                  │
  │              │ 내부  │  내부 스키마 (1개)│
  │              └───┬───┘                  │
  │              ┌───▼───┐                  │
  │              │ DB    │  물리적 저장소    │
  │              └───────┘                  │
  └─────────────────────────────────────────┘
```

#### 외부 스키마 (External Schema = 사용자 뷰)

```
- 사용자/응용 프로그래머 관점의 DB 구조
- 여러 개 존재 가능
- 각 사용자마다 필요한 부분만 봄
- = 서브 스키마 = 사용자 뷰

예시: 같은 DB인데...
  학생은 "수강신청 화면"만 봄 (외부 스키마 1)
  교수는 "성적입력 화면"만 봄 (외부 스키마 2)
  관리자는 "전체관리 화면"을 봄 (외부 스키마 3)
```

#### 개념 스키마 (Conceptual Schema)

```
- 전체 DB의 논리적 구조
- 딱 1개만 존재
- 모든 사용자가 공유하는 통합된 구조
- 일반적으로 "스키마"라 하면 개념 스키마를 의미
- 개체, 관계, 제약조건 등 포함

예시: 대학교 DB 전체 구조
  학생(학번, 이름, 학과코드)
  학과(학과코드, 학과명)
  과목(과목코드, 과목명)
  수강(학번, 과목코드, 성적)
```

#### 내부 스키마 (Internal Schema)

```
- 물리적 저장 구조
- 딱 1개만 존재
- 실제 데이터가 어떻게 저장되는지 정의
- 인덱스, 레코드 크기, 저장 경로 등

예시:
  - 학생 테이블은 B+트리 인덱스 사용
  - 레코드 크기: 100바이트
  - 저장 위치: /data/student.dbf
```

### 5.2 데이터 독립성

```
┌──────────────────────────────────────────────────┐
│ 논리적 데이터 독립성                              │
│   외부 스키마 ←→ 개념 스키마 사이                 │
│   개념 스키마 변경해도 외부 스키마 영향 없음      │
│   (외부/개념 사상에 의해 보장)                    │
├──────────────────────────────────────────────────┤
│ 물리적 데이터 독립성                              │
│   개념 스키마 ←→ 내부 스키마 사이                 │
│   내부 스키마 변경해도 개념 스키마 영향 없음      │
│   (개념/내부 사상에 의해 보장)                    │
└──────────────────────────────────────────────────┘
```

### 6.0 SQL 분류

### 6.1 SELECT 완전정복

#### 기본 구조

```sql
SELECT [DISTINCT] 컬럼1, 컬럼2, ...    -- 조회할 컬럼
FROM 테이블명                           -- 대상 테이블
[WHERE 조건]                            -- 행 필터링
[GROUP BY 컬럼]                         -- 그룹화
[HAVING 그룹조건]                       -- 그룹 필터링
[ORDER BY 컬럼 [ASC|DESC]]             -- 정렬
```

```
★ 실행 순서 (암기 필수!):
  FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY

★ 암기법: "프(FROM) 웨(WHERE) 그(GROUP) 해(HAVING) 셀(SELECT) 오(ORDER)"
```

#### WHERE 조건절

```sql
-- 비교 연산자
SELECT * FROM 학생 WHERE 학년 = 3;
SELECT * FROM 학생 WHERE 학년 >= 2;
SELECT * FROM 학생 WHERE 학년 <> 3;          -- 같지 않다 (!=도 가능)

-- 논리 연산자
SELECT * FROM 학생 WHERE 학과='컴공' AND 학년=3;
SELECT * FROM 학생 WHERE 학과='컴공' OR 학과='전자';

-- BETWEEN
SELECT * FROM 성적 WHERE 점수 BETWEEN 80 AND 100;  -- 80 이상 100 이하

-- IN / NOT IN
SELECT * FROM 학생 WHERE 학과 IN ('컴공', '전자', '수학');
SELECT * FROM 학생 WHERE 학과 NOT IN ('컴공');

-- LIKE (패턴 매칭)
SELECT * FROM 학생 WHERE 이름 LIKE '김%';     -- '김'으로 시작
SELECT * FROM 학생 WHERE 이름 LIKE '%수';     -- '수'로 끝남
SELECT * FROM 학생 WHERE 이름 LIKE '_영_';    -- 두번째 글자가 '영'
-- %: 0개 이상 문자, _: 정확히 1개 문자

-- IS NULL / IS NOT NULL
SELECT * FROM 학생 WHERE 전화번호 IS NULL;
SELECT * FROM 학생 WHERE 전화번호 IS NOT NULL;

★ 주의: NULL 비교는 = 이 아니라 IS NULL 사용!
  WHERE 전화번호 = NULL  → (X) 틀린 문법!
  WHERE 전화번호 IS NULL → (O) 올바른 문법!
```

#### 집계 함수

```sql
-- COUNT: 행의 수
SELECT COUNT(*) FROM 학생;                    -- 전체 행 수
SELECT COUNT(학과) FROM 학생;                 -- NULL 제외 행 수
SELECT COUNT(DISTINCT 학과) FROM 학생;        -- 중복 제거 후 행 수

-- SUM, AVG, MIN, MAX
SELECT SUM(점수) FROM 성적;
SELECT AVG(점수) FROM 성적;
SELECT MIN(점수) FROM 성적;
SELECT MAX(점수) FROM 성적;

★ 주의: COUNT(*)는 NULL 포함, COUNT(컬럼명)은 NULL 제외!
```

#### GROUP BY + HAVING

```sql
-- 학과별 학생 수
SELECT 학과, COUNT(*) AS 학생수
FROM 학생
GROUP BY 학과;

-- 학과별 학생수가 3명 이상인 학과
SELECT 학과, COUNT(*) AS 학생수
FROM 학생
GROUP BY 학과
HAVING COUNT(*) >= 3;

-- 학과별 평균 점수가 80 이상인 학과의 평균 점수
SELECT 학과, AVG(점수) AS 평균점수
FROM 성적 NATURAL JOIN 학생
GROUP BY 학과
HAVING AVG(점수) >= 80;
```

```
★ WHERE vs HAVING 차이:
  WHERE:  GROUP BY 이전에 적용 (개별 행 필터링)
  HAVING: GROUP BY 이후에 적용 (그룹 필터링, 집계 함수 사용 가능)

★ 기출 함정:
  WHERE COUNT(*) >= 3   → (X) WHERE에서 집계함수 사용 불가!
  HAVING COUNT(*) >= 3  → (O) HAVING에서 집계함수 사용!
```

#### ORDER BY

```sql
-- 오름차순 (기본값, ASC 생략 가능)
SELECT * FROM 학생 ORDER BY 이름;
SELECT * FROM 학생 ORDER BY 이름 ASC;

-- 내림차순
SELECT * FROM 학생 ORDER BY 점수 DESC;

-- 복합 정렬 (1차: 학과 오름차순, 2차: 이름 내림차순)
SELECT * FROM 학생 ORDER BY 학과 ASC, 이름 DESC;

★ ASC = 오름차순 (Ascending) = 작은 값 → 큰 값 (1, 2, 3...)
★ DESC = 내림차순 (Descending) = 큰 값 → 작은 값 (3, 2, 1...)
★ 기본값은 ASC (생략하면 오름차순)
```

#### JOIN (조인)

```sql
-- [방법1] WHERE절 조인 (구형)
SELECT 학생.이름, 학과.학과명
FROM 학생, 학과
WHERE 학생.학과코드 = 학과.학과코드;

-- [방법2] INNER JOIN (표준, 권장)
SELECT 학생.이름, 학과.학과명
FROM 학생 INNER JOIN 학과
ON 학생.학과코드 = 학과.학과코드;

-- [방법3] NATURAL JOIN (동일 컬럼명 자동 조인, 중복 컬럼 제거)
SELECT 이름, 학과명
FROM 학생 NATURAL JOIN 학과;

-- LEFT (OUTER) JOIN: 왼쪽 테이블 기준, 매칭 없으면 NULL
SELECT 학생.이름, 학과.학과명
FROM 학생 LEFT JOIN 학과
ON 학생.학과코드 = 학과.학과코드;

-- RIGHT (OUTER) JOIN: 오른쪽 테이블 기준
SELECT 학생.이름, 학과.학과명
FROM 학생 RIGHT JOIN 학과
ON 학생.학과코드 = 학과.학과코드;
```

```
★ JOIN 정리:
  INNER JOIN: 양쪽 모두 매칭되는 행만
  LEFT JOIN:  왼쪽 전부 + 오른쪽 매칭 (없으면 NULL)
  RIGHT JOIN: 오른쪽 전부 + 왼쪽 매칭 (없으면 NULL)
  FULL JOIN:  양쪽 전부 (없으면 NULL)
  NATURAL JOIN: 동일 컬럼명 자동 조인 + 중복 컬럼 제거
```

#### 서브쿼리 (Subquery)

```sql
-- WHERE절 서브쿼리: 조건에 다른 쿼리 결과 사용
-- "컴공 학과에 속한 학생들의 이름"
SELECT 이름
FROM 학생
WHERE 학과코드 = (SELECT 학과코드 FROM 학과 WHERE 학과명 = '컴공');

-- IN 서브쿼리: 여러 값 비교
-- "DB 과목을 수강한 학생들의 이름"
SELECT 이름
FROM 학생
WHERE 학번 IN (SELECT 학번 FROM 수강 WHERE 과목명 = 'DB');

-- FROM절 서브쿼리 (인라인 뷰)
-- "학과별 평균 점수 테이블에서 80점 이상인 학과"
SELECT 학과, 평균점수
FROM (SELECT 학과, AVG(점수) AS 평균점수
      FROM 성적
      GROUP BY 학과) AS T
WHERE 평균점수 >= 80;

★ 서브쿼리 결과가 1개 → = 사용
★ 서브쿼리 결과가 여러 개 → IN 사용
```

#### 실전 예제 5선

```
예제 데이터:

학생 테이블:
┌──────┬───────┬──────────┐
│ 학번 │ 이름  │ 학과코드 │
├──────┼───────┼──────────┤
│ 100  │ 김철수│   CS     │
│ 200  │ 이영희│   EE     │
│ 300  │ 박민수│   CS     │
│ 400  │ 정다은│   MA     │
└──────┴───────┴──────────┘

성적 테이블:
┌──────┬──────────┬──────┐
│ 학번 │ 과목코드 │ 점수 │
├──────┼──────────┼──────┤
│ 100  │   D001   │  90  │
│ 100  │   D002   │  85  │
│ 200  │   D001   │  78  │
│ 300  │   D001   │  92  │
│ 300  │   D003   │  88  │
└──────┴──────────┴──────┘
```

```sql
-- [예제1] 학과코드가 'CS'인 학생의 이름 조회
SELECT 이름
FROM 학생
WHERE 학과코드 = 'CS';
-- 결과: 김철수, 박민수

-- [예제2] 학번별 총점과 평균 점수 조회 (소수점 이하 반올림)
SELECT 학번, SUM(점수) AS 총점, ROUND(AVG(점수), 1) AS 평균
FROM 성적
GROUP BY 학번;
-- 결과: 100|175|87.5, 200|78|78.0, 300|180|90.0

-- [예제3] 2과목 이상 수강한 학생의 학번과 수강 과목 수
SELECT 학번, COUNT(*) AS 과목수
FROM 성적
GROUP BY 학번
HAVING COUNT(*) >= 2;
-- 결과: 100|2, 300|2

-- [예제4] 학생 이름과 점수 조회 (성적 없는 학생도 포함)
SELECT 학생.이름, 성적.점수
FROM 학생 LEFT JOIN 성적
ON 학생.학번 = 성적.학번;
-- 결과: 김철수|90, 김철수|85, 이영희|78, 박민수|92, 박민수|88, 정다은|NULL

-- [예제5] 과목코드 'D001'의 최고 점수 학생 이름
SELECT 이름
FROM 학생
WHERE 학번 = (SELECT 학번 FROM 성적
              WHERE 과목코드 = 'D001'
              ORDER BY 점수 DESC
              LIMIT 1);
-- 또는
SELECT 이름
FROM 학생
WHERE 학번 IN (SELECT 학번 FROM 성적
               WHERE 과목코드 = 'D001'
               AND 점수 = (SELECT MAX(점수) FROM 성적
                           WHERE 과목코드 = 'D001'));
-- 결과: 박민수 (92점)
```

```sql
-- 기본 구조
```

### 6.4 DELETE (삭제)

```sql
-- 기본 구조
DELETE FROM 테이블명
WHERE 조건;

-- 예시: 학번 400 학생 삭제
DELETE FROM 학생
WHERE 학번 = 400;

-- 학과코드가 'MA'인 학생 모두 삭제
DELETE FROM 학생
WHERE 학과코드 = 'MA';

★ 주의: WHERE 절 없으면 전체 행 삭제!
  DELETE FROM 학생;  -- 학생 테이블의 모든 데이터 삭제 (테이블 구조는 유지)

★ DELETE vs DROP vs TRUNCATE 비교:
  DELETE:   DML, 행 삭제, 조건 가능, ROLLBACK 가능
  DROP:     DDL, 테이블 자체 삭제, ROLLBACK 불가
  TRUNCATE: DDL, 전체 행 삭제(구조 유지), ROLLBACK 불가, DELETE보다 빠름
```

### 6.5 DDL (CREATE, ALTER, DROP)

#### CREATE TABLE

```sql
CREATE TABLE 학생 (
    학번 INT PRIMARY KEY,
    이름 VARCHAR(20) NOT NULL,
    학과코드 VARCHAR(5),
    학년 INT DEFAULT 1,
    FOREIGN KEY(학과코드) REFERENCES 학과(학과코드)
);

-- 주요 제약조건:
--   PRIMARY KEY: 기본키
--   FOREIGN KEY ... REFERENCES: 외래키
--   NOT NULL: NULL 불허
--   UNIQUE: 중복 불허 (NULL은 허용)
--   DEFAULT: 기본값
--   CHECK: 조건 제약 (예: CHECK(학년 BETWEEN 1 AND 4))
```

#### ALTER TABLE

```sql
-- 컬럼 추가 (ADD)
ALTER TABLE 학생 ADD 전화번호 VARCHAR(15);

-- 컬럼 수정 (MODIFY / ALTER COLUMN)
ALTER TABLE 학생 MODIFY 이름 VARCHAR(30);

-- 컬럼 삭제 (DROP COLUMN)
ALTER TABLE 학생 DROP COLUMN 전화번호;

★ 암기: ALTER = 변경 (ADD 추가, MODIFY 수정, DROP 삭제)
```

#### DROP (삭제)

```sql
-- 테이블 삭제
DROP TABLE 학생;

-- 뷰 삭제 (시험 빈출!)
DROP VIEW 학생뷰 CASCADE;    -- 연관된 모든 객체도 함께 삭제
DROP VIEW 학생뷰 RESTRICT;   -- 연관된 객체 있으면 삭제 거부

★ CASCADE vs RESTRICT (초빈출!):
  CASCADE:  연쇄 삭제 (참조하는 것들도 다 삭제)
  RESTRICT: 제한 삭제 (참조하는 것이 있으면 삭제 안 함)

★ 암기:
  CASCADE = "캐스캐이드 = 폭포수 = 연쇄적으로 다 삭제"
  RESTRICT = "리스트릭트 = 제한 = 참조 있으면 삭제 거부"
```

### 🔥 SQL 빈칸 채우기 패턴 총정리

```
★ 시험에서 가장 많이 나오는 빈칸 위치:

[패턴1] SELECT 빈칸
  SELECT (   ) FROM 학생 WHERE 학과='CS';
  → 컬럼명 또는 * 또는 집계함수

[패턴2] FROM 빈칸
  SELECT 이름 FROM (   ) WHERE 학번=100;
  → 테이블명

[패턴3] WHERE 빈칸
  SELECT * FROM 학생 WHERE 학과 (   ) ('CS','EE');
  → IN

[패턴4] JOIN 빈칸
  SELECT * FROM 학생 (   ) JOIN 학과 ON 학생.학과코드=학과.학과코드;
  → INNER / LEFT / RIGHT / NATURAL (문맥에 따라)

[패턴5] INSERT 빈칸
  INSERT (   ) 학생 (   ) (100, '김철수', 'CS');
  → INTO, VALUES

[패턴6] UPDATE 빈칸
  UPDATE 학생 (   ) 이름='홍길동' (   ) 학번=100;
  → SET, WHERE

[패턴7] GROUP BY + HAVING 빈칸
  SELECT 학과, COUNT(*)
  FROM 학생
  (   ) 학과
  (   ) COUNT(*) >= 3;
  → GROUP BY, HAVING

[패턴8] ORDER BY 빈칸
  SELECT * FROM 학생 ORDER BY 이름 (   );
  → ASC 또는 DESC

[패턴9] DROP 빈칸
  DROP VIEW 학생뷰 (   );
  → CASCADE 또는 RESTRICT

[패턴10] 서브쿼리 빈칸
  SELECT 이름 FROM 학생
  WHERE 학번 (   ) (SELECT 학번 FROM 수강 WHERE 과목='DB');
  → IN
```

## 7장. 관계대수 ★★☆

> **출제 빈도: 5/9회 | 난이도: 중 | 기호-의미 매칭 문제**

### 7.1 관계대수란?

```
정의: 릴레이션을 처리하는 연산의 집합
      원하는 데이터를 얻기 위해 "어떻게" 유도하는지 기술하는 절차적 언어

종류:
  (1) 순수 관계 연산자: SELECT(σ), PROJECT(π), JOIN(⋈), DIVISION(÷)
  (2) 일반 집합 연산자: UNION(∪), INTERSECTION(∩), DIFFERENCE(-), CARTESIAN PRODUCT(×)
```

### 7.2 기호와 의미 총정리

| 기호 | 이름 | 의미 | SQL 대응 |
|:---:|:---:|:---:|:---:|
| **σ** (시그마) | SELECT (셀렉트) | **행** 선택 (조건에 맞는 튜플) | WHERE |
| **π** (파이) | PROJECT (프로젝트) | **열** 선택 (원하는 속성만) | SELECT 컬럼 |
| **⋈** | JOIN (조인) | 두 릴레이션 결합 | JOIN |
| **÷** | DIVISION (디비전) | 나눗셈 연산 | - |
| **∪** | UNION (합집합) | 합집합 | UNION |
| **∩** | INTERSECTION (교집합) | 교집합 | INTERSECT |
| **-** | DIFFERENCE (차집합) | 차집합 | EXCEPT/MINUS |
| **×** | CARTESIAN PRODUCT | 모든 조합 | CROSS JOIN |

```
★ 가장 헷갈리는 것: σ(SELECT) vs π(PROJECT)

  σ = 행(가로) 선택 = WHERE 조건 → "시그마는 가로(행) 선택"
  π = 열(세로) 선택 = 컬럼 선택 → "파이는 세로(열) 선택"

★ 암기법:
  "시(σ)행(행 선택)" = 시그마는 행 선택
  "파(π)열(열 선택)" = 파이는 열 선택

  또는: σ(시그마)의 가운데 가로줄 → 가로(행) 선택
        π의 세로 기둥 → 세로(열) 선택
```

### 7.3 셀렉트(σ)와 프로젝트(π) 상세

#### σ (SELECT / 셀렉트) - 행 선택

```
표기: σ조건(R)

의미: 릴레이션 R에서 조건을 만족하는 튜플(행)을 선택

예시:
  학생 릴레이션에서 학과가 '컴공'인 학생 선택
  → σ학과='컴공'(학생)

  SQL 대응: SELECT * FROM 학생 WHERE 학과='컴공';

  학생:
  ┌──────┬───────┬──────┐        σ학과='컴공'(학생):
  │ 학번 │ 이름  │ 학과 │        ┌──────┬───────┬──────┐
  │ 100  │ 김철수│ 컴공 │  ─→    │ 100  │ 김철수│ 컴공 │
  │ 200  │ 이영희│ 전자 │        │ 300  │ 박민수│ 컴공 │
  │ 300  │ 박민수│ 컴공 │        └──────┴───────┴──────┘
  └──────┴───────┴──────┘
```

#### π (PROJECT / 프로젝트) - 열 선택

```
표기: π속성리스트(R)

의미: 릴레이션 R에서 원하는 속성(열)만 추출 (중복 제거)

예시:
  학생 릴레이션에서 이름만 추출
  → π이름(학생)

  SQL 대응: SELECT DISTINCT 이름 FROM 학생;

  학생:
  ┌──────┬───────┬──────┐        π이름(학생):
  │ 학번 │ 이름  │ 학과 │        ┌───────┐
  │ 100  │ 김철수│ 컴공 │  ─→    │ 김철수│
  │ 200  │ 이영희│ 전자 │        │ 이영희│
  │ 300  │ 박민수│ 컴공 │        │ 박민수│
  └──────┴───────┴──────┘        └───────┘

★ PROJECT는 중복 튜플을 자동 제거!
```

#### 복합 사용 예시

```
"컴공 학과 학생의 이름만 조회"

  → π이름(σ학과='컴공'(학생))

  풀이:
  1단계: σ학과='컴공'(학생) → 컴공 학과 행만 선택
  2단계: π이름(결과)       → 이름 열만 추출

  SQL: SELECT DISTINCT 이름 FROM 학생 WHERE 학과='컴공';
```

### 7.4 조인(⋈)과 디비전(÷)

#### ⋈ (JOIN / 조인)

```
표기: R ⋈조건 S

의미: 두 릴레이션을 조건에 따라 결합

예시:
  학생 ⋈학생.학과코드=학과.학과코드 학과
  → 학과코드가 같은 학생과 학과를 결합

  SQL: SELECT * FROM 학생 JOIN 학과 ON 학생.학과코드=학과.학과코드;

자연 조인 (Natural Join):
  R ⋈ S  (조건 없이 동일 속성으로 자동 조인, 중복 속성 제거)
```

#### ÷ (DIVISION / 디비전)

```
표기: R ÷ S

의미: R에서 S의 모든 값을 가진 튜플의 나머지 속성

예시:
  수강 릴레이션:                  과목 릴레이션:
  ┌──────┬──────┐               ┌──────┐
  │ 학번 │ 과목 │               │ 과목 │
  ├──────┼──────┤               ├──────┤
  │ 100  │  DB  │               │  DB  │
  │ 100  │  OS  │               │  OS  │
  │ 200  │  DB  │               └──────┘
  │ 300  │  DB  │
  │ 300  │  OS  │               수강 ÷ 과목:
  └──────┴──────┘               ┌──────┐
                                │ 학번 │
                                ├──────┤
                                │ 100  │  ← DB와 OS 모두 수강
                                │ 300  │  ← DB와 OS 모두 수강
                                └──────┘
  (학번 200은 DB만 수강 → 제외)

★ 핵심: "S의 모든 값을 가진 행만 추출"
★ 디비전은 "모든 과목을 수강한 학생" 같은 문제에 사용
```

### 7.5 집합 연산 (∪, ∩, -)

```
전제: 두 릴레이션의 차수(Degree)가 같고, 대응 속성의 도메인이 같아야 함

  R:                    S:
  ┌──────┬───────┐     ┌──────┬───────┐
  │ 학번 │ 이름  │     │ 학번 │ 이름  │
  ├──────┼───────┤     ├──────┼───────┤
  │ 100  │ 김철수│     │ 100  │ 김철수│
  │ 200  │ 이영희│     │ 300  │ 박민수│
  │ 300  │ 박민수│     │ 400  │ 정다은│
  └──────┴───────┘     └──────┴───────┘

  R ∪ S (합집합):       R ∩ S (교집합):       R - S (차집합):
  ┌──────┬───────┐     ┌──────┬───────┐     ┌──────┬───────┐
  │ 100  │ 김철수│     │ 100  │ 김철수│     │ 200  │ 이영희│
  │ 200  │ 이영희│     │ 300  │ 박민수│     └──────┴───────┘
  │ 300  │ 박민수│     └──────┴───────┘     (R에만 있고 S에 없는 것)
  │ 400  │ 정다은│
  └──────┴───────┘
```

## 8장. 조인 종류 ★★☆

> **출제 빈도: 3/9회 | 난이도: 중 | 용어 비교 문제**

### 8.1 조인 종류 총정리

```
┌─────────────────────────────────────────────────────────┐
│                    조인의 분류                           │
│                                                         │
│  ┌── 세타 조인 (Theta Join): 비교 연산자 사용          │
│  │    └── 동등 조인 (Equi Join): = 연산자만 사용       │
│  │         └── 자연 조인 (Natural Join): 중복 속성 제거│
│  │                                                      │
│  ├── 외부 조인 (Outer Join): 매칭 안되는 행도 포함     │
│  │    ├── LEFT OUTER JOIN                               │
│  │    ├── RIGHT OUTER JOIN                              │
│  │    └── FULL OUTER JOIN                               │
│  │                                                      │
│  ├── 셀프 조인 (Self Join): 자기 자신과 조인           │
│  │                                                      │
│  └── 카티션 프로덕트 (Cartesian Product): 모든 조합    │
│       = CROSS JOIN                                      │
└─────────────────────────────────────────────────────────┘
```

### 8.2 각 조인 상세

#### 세타 조인 (Theta Join, θ-Join)

```
정의: 비교 연산자(=, <, >, <=, >=, <>)를 조건으로 사용하는 조인
표기: R ⋈θ조건 S

예시: 학생 ⋈학생.학년 >= 과목.최소학년 과목
  → 학생의 학년이 과목의 최소학년 이상인 조합

★ 세타 조인이 가장 일반적인 형태 (상위 개념)
```

#### 동등 조인 (Equi Join)

```
정의: 세타 조인 중 = (같다) 조건만 사용하는 조인
표기: R ⋈R.A=S.B S

예시: 학생 ⋈학생.학과코드=학과.학과코드 학과
  → 학과코드가 같은 것끼리 결합

★ 동등 조인 = 세타 조인의 특수한 경우 (= 만 사용)
★ 중복 속성(학과코드)이 양쪽 모두에 나타남!
```

#### 자연 조인 (Natural Join)

```
정의: 동등 조인에서 중복 속성을 제거한 조인
표기: R ⋈ S (조건 없이 같은 이름의 속성으로 자동 조인)

예시: 학생 ⋈ 학과
  → 동일 속성명(학과코드)으로 자동 조인 + 학과코드 중복 제거

  동등 조인 결과:                    자연 조인 결과:
  ┌──────┬──────────┬──────────┬──────┐  ┌──────┬──────────┬──────┐
  │ 학번 │학과코드  │학과코드  │학과명│  │ 학번 │학과코드  │학과명│
  │      │(학생)    │(학과)    │      │  │      │          │      │
  ├──────┼──────────┼──────────┼──────┤  ├──────┼──────────┼──────┤
  │ 100  │   CS     │   CS     │ 컴공 │  │ 100  │   CS     │ 컴공 │
  └──────┴──────────┴──────────┴──────┘  └──────┴──────────┴──────┘
  ↑ 학과코드 2개 (중복)                  ↑ 학과코드 1개 (중복 제거!)

★ 자연 조인 = 동등 조인 - 중복 속성
★ 가장 많이 사용되는 조인!
```

#### 외부 조인 (Outer Join)

```
정의: 조인 조건에 맞지 않는 튜플도 결과에 포함 (NULL로 채움)

LEFT OUTER JOIN: 왼쪽 릴레이션의 모든 튜플 포함
RIGHT OUTER JOIN: 오른쪽 릴레이션의 모든 튜플 포함
FULL OUTER JOIN: 양쪽 릴레이션의 모든 튜플 포함

예시: 학생 LEFT JOIN 성적
  → 성적이 없는 학생도 결과에 포함 (성적 부분은 NULL)
```

#### 셀프 조인 (Self Join)

```
정의: 같은 릴레이션(자기 자신)과 조인

예시: 사원 테이블에서 상사의 이름을 찾을 때
  사원(사번, 이름, 상사사번)

  SELECT A.이름 AS 사원, B.이름 AS 상사
  FROM 사원 A JOIN 사원 B
  ON A.상사사번 = B.사번;
```

#### 카티션 프로덕트 (Cartesian Product)

```
정의: 두 릴레이션의 모든 가능한 조합 (조건 없는 결합)
표기: R × S

결과 크기:
  Degree = R의 Degree + S의 Degree
  Cardinality = R의 Cardinality × S의 Cardinality

예시: R(3행) × S(4행) = 12행

★ 카티션 프로덕트는 실수로 WHERE 없이 조인하면 발생!
★ SQL: SELECT * FROM R, S;  (WHERE 없음)
★ SQL: SELECT * FROM R CROSS JOIN S;
```

### 🔥 조인 비교표

| 조인 | 조건 | 중복 속성 | 특징 |
|:---:|:---:|:---:|:---:|
| 세타 조인 | 비교 연산자 전체 | 있음 | 가장 일반적 |
| 동등 조인 | = 만 사용 | 있음 | 세타의 특수 |
| 자연 조인 | 동일 속성 자동 | **제거** | 동등+중복제거 |
| 외부 조인 | 매칭 없어도 포함 | - | NULL 채움 |
| 셀프 조인 | 자기 자신과 | - | 별칭 필수 |
| 카티션 | 조건 없음 | 있음 | 모든 조합 |

```
★ 핵심 관계:
  세타 조인 ⊃ 동등 조인 ⊃ 자연 조인
  "세타가 제일 넓고, 자연이 제일 좁다(조건 엄격)"
```

## 9장. DB 설계 순서 ★★☆

> **출제 빈도: 3/9회 | 난이도: 하 | 순서 나열 문제**

### 9.1 데이터베이스 설계 5단계

```
┌──────────────┐
│ 요구조건 분석 │  → 사용자 요구사항 수집, 분석
├──────────────┤
│ 개념적 설계   │  → E-R 다이어그램 작성, 개념 스키마
├──────────────┤
│ 논리적 설계   │  → 릴레이션 스키마 변환, 정규화
├──────────────┤
│ 물리적 설계   │  → 저장 구조, 인덱스, 접근 경로 설계
├──────────────┤
│ 구현          │  → DDL로 스키마 생성, 데이터 적재
└──────────────┘
```

```
★ 암기법: "요개논물구" (요리 개발 논문 물어 구함)
  요(구조건분석) → 개(념적설계) → 논(리적설계) → 물(리적설계) → 구(현)
```

### 9.2 각 단계 상세

| 단계 | 핵심 활동 | 산출물 | 키워드 |
|:---:|:---:|:---:|:---:|
| 요구조건 분석 | 사용자 인터뷰, 요구사항 수집 | 요구사항 명세서 | 무엇을 원하는가? |
| 개념적 설계 | E-R 다이어그램, 개념 스키마 | 개념적 스키마 | 엔터티, 관계 |
| 논리적 설계 | 릴레이션 변환, 정규화 | 논리적 스키마 | 테이블, 정규화 |
| 물리적 설계 | 저장 구조, 인덱스, 해싱 | 물리적 스키마 | 성능, 저장 |
| 구현 | DDL 작성, 트랜잭션 | 데이터베이스 | SQL, 코딩 |

### 9.3 물리적 설계 - 접근 방법

```
(1) 순차 접근 (Sequential Access)
    - 순서대로 접근
    - 전체 검색 시 효율적

(2) 인덱스 (Index)
    - 검색 속도 향상을 위한 자료구조
    - B-트리, B+트리 등
    - 특정 레코드 빠르게 찾기

(3) 해싱 (Hashing)
    - 해시 함수로 저장 위치 직접 계산
    - 등가 검색(=)에 매우 빠름
    - 범위 검색에는 부적합
```

### 🎯 기출 확인 문제

```
  구현, 논리적 설계, 물리적 설계, 요구조건 분석, 개념적 설계

[정답]
  요구조건 분석 → 개념적 설계 → 논리적 설계 → 물리적 설계 → 구현
```

```
(2) 대체키: {C}
```

### [예상문제 6] ★★★ SQL SELECT

```
다음 SQL의 실행 결과를 쓰시오.

  ┌──────┬──────────┬──────┐
  │ 학번 │   과목   │ 점수 │
  ├──────┼──────────┼──────┤
  │ 100  │   DB     │  90  │
  │ 100  │   OS     │  85  │
  │ 200  │   DB     │  78  │
  │ 200  │   OS     │  92  │
  │ 300  │   DB     │  65  │
  └──────┴──────────┴──────┘

  GROUP BY 학번
  HAVING AVG(점수) >= 80;
```

```
│ 200  │ 85.0 │
└──────┴──────┘

학번 300: 65/1 = 65.0 → 80 미만 → 제외
```

## 부록. 최종 암기 치트시트

> **시험장 입실 전 마지막으로 보는 페이지!**

### A. 용어 한줄 정리

```
Degree        = 속성(컬럼) 수
Cardinality   = 튜플(행) 수
Tuple         = 행 = Row = Record
Attribute     = 열 = Column = Field
Domain        = 속성 값의 범위
Schema        = 릴레이션 구조 정의
Instance      = 특정 시점의 실제 데이터
```

### B. 키 한줄 정리

```
슈퍼키   = 유일성 O, 최소성 X
후보키   = 유일성 O, 최소성 O
기본키   = 후보키 중 대표 1개 (NULL X, 중복 X)
대체키   = 후보키 - 기본키
외래키   = 다른 테이블 PK 참조 (NULL O)
```

### C. 무결성 한줄 정리

```
개체 무결성  = PK → NULL 불가, 중복 불가
참조 무결성  = FK → 참조 PK 값 or NULL
도메인 무결성 = 속성 값 → 도메인 범위 내
```

### D. 정규화 한줄 정리

```
1NF  = 원자값 (다중값 X)
2NF  = 부분 함수 종속 제거
3NF  = 이행 함수 종속 제거
BCNF = 모든 결정자가 후보키
암기: "도부이결" (도메인원자 / 부분제거 / 이행제거 / 결정자=후보키)
```

### E. 스키마 한줄 정리

```
외부 스키마 = 사용자 관점, 여러 개
개념 스키마 = 전체 논리 구조, 1개
내부 스키마 = 물리적 저장, 1개
```

### F. 관계대수 한줄 정리

```
σ(시그마)  = SELECT  = 행 선택 (WHERE)
π(파이)    = PROJECT = 열 선택 (SELECT 컬럼)
⋈          = JOIN    = 결합
÷          = DIVISION= 나눗셈 (모든 값 포함)
∪          = UNION   = 합집합
∩          = INTERSECT= 교집합
-          = MINUS   = 차집합
×          = CROSS   = 카티션 프로덕트
```

### G. SQL 핵심 키워드

```
SELECT  FROM  WHERE  GROUP BY  HAVING  ORDER BY
INSERT INTO ... VALUES / INSERT INTO ... SELECT
UPDATE ... SET ... WHERE
DELETE FROM ... WHERE
DROP VIEW ... CASCADE / RESTRICT
```

### H. DB 설계 순서

```
요구조건 분석 → 개념적 설계 → 논리적 설계 → 물리적 설계 → 구현
"요개논물구"
```

### I. 조인 계층

```
세타 조인 ⊃ 동등 조인 ⊃ 자연 조인
자연 조인 = 동등 조인 - 중복 속성
```
