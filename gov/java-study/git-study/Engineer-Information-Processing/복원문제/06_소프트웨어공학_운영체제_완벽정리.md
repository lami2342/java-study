## 1장. 테스트 커버리지 (Test Coverage) ★★★ - 매회 출제!

### 1.1 커버리지란?

테스트가 소스코드를 **얼마나 충분히 검증했는가**를 나타내는 척도.
"이 테스트로 코드의 몇 %를 확인했는가?"

---

### 1.2 커버리지 강도 순서 (약 → 강)

```
문장 < 분기(결정) < 조건 < 조건/결정 < 변경조건/결정(MC/DC) < 다중조건 < 경로
```

**암기법: "문분조 조변다경"** (문 분 조 / 조 변 다 경)
- **문**장 → **분**기 → **조**건 → **조**건결정 → **변**경조건결정 → **다**중조건 → **경**로

---

### 1.3 각 커버리지 상세 설명

#### (1) 문장 커버리지 (Statement Coverage)
- **정의**: 모든 **실행 문장**을 최소 **1번 이상** 실행
- **가장 약한** 커버리지
- 분기를 고려하지 않음

```
예시 코드:
  1: x = 10
  2: if (x > 5)
  3:     print("크다")    ← 이 문장 실행 필요
  4: else
  5:     print("작다")    ← 이 문장도 실행 필요
  6: print("끝")

문장 커버리지 100% = 모든 문장(1,2,3,5,6)이 최소 1번 실행
→ 테스트케이스 2개 필요: x=10 (1,2,3,6 실행), x=3 (1,2,5,6 실행)
```

#### (2) 분기 커버리지 (Branch/Decision Coverage)
- **정의**: 모든 **분기(참/거짓)**를 최소 **1번씩** 실행
- = 결정 커버리지 (Decision Coverage)

```
예시:
  if (A > 0)       → 참 1번, 거짓 1번 필요
      처리1
  else
      처리2

분기 커버리지 100% = 모든 if문의 참/거짓 경로 실행
```

#### (3) 조건 커버리지 (Condition Coverage) ★★★ 초빈출!
- **정의**: 각 개별 **조건식**이 **참(T)과 거짓(F)을 모두** 갖도록 테스트
- **주의**: 전체 결정(Decision)의 T/F는 보장하지 않을 수 있음!

```
예시: if (A > 0 AND B > 0)
  조건1: A > 0  → T, F 모두 필요
  조건2: B > 0  → T, F 모두 필요

테스트케이스:
  TC1: A=5(T), B=-1(F) → 전체 결정: F
  TC2: A=-1(F), B=5(T) → 전체 결정: F

→ 각 조건은 T/F 모두 가졌지만, 전체 결정은 F만 나옴!
→ 조건 커버리지 100%이지만 분기 커버리지 100%는 아닐 수 있다!
```

**시험 포인트**: "조건 커버리지가 분기 커버리지를 만족시키는가?" → **아니다!**

#### (4) 조건/결정 커버리지 (Condition/Decision Coverage)
- **정의**: **조건 커버리지 + 분기(결정) 커버리지** 동시 만족
- 각 조건이 T/F + 전체 결정도 T/F

```
예시: if (A > 0 AND B > 0)

테스트케이스:
  TC1: A=5(T), B=5(T) → 전체 결정: T  ← 결정의 T 확보
  TC2: A=-1(F), B=-1(F) → 전체 결정: F  ← 결정의 F 확보

→ 조건1(A>0): T,F 모두 / 조건2(B>0): T,F 모두 / 결정: T,F 모두 ✓
```

#### (5) 변경 조건/결정 커버리지 (MC/DC: Modified Condition/Decision Coverage)
- **정의**: 각 개별 조건이 **독립적으로** 전체 결정 결과에 **영향을 미치는** 것을 증명
- 항공/국방 등 고신뢰성 소프트웨어에서 사용

```
예시: if (A AND B)

  TC1: A=T, B=T → 결정=T
  TC2: A=F, B=T → 결정=F   ← A만 변경 → 결정 변경 (A의 독립적 영향 증명)
  TC3: A=T, B=F → 결정=F   ← B만 변경 → 결정 변경 (B의 독립적 영향 증명)

→ 3개의 테스트케이스로 MC/DC 달성
```

#### (6) 다중 조건 커버리지 (Multiple Condition Coverage)
- **정의**: 모든 조건의 **가능한 조합**을 전부 테스트
- 조건이 N개면 최대 **2^N개** 테스트케이스

```
예시: if (A AND B)  → 조건 2개 → 2^2 = 4가지 조합

  TC1: A=T, B=T → T
  TC2: A=T, B=F → F
  TC3: A=F, B=T → F
  TC4: A=F, B=F → F

→ 모든 조합을 다 테스트
```

#### (7) 경로 커버리지 (Path Coverage)
- **정의**: 프로그램의 **모든 가능한 실행 경로**를 테스트
- **가장 강한** 커버리지
- 반복문이 있으면 경로가 무한대 → 현실적으로 달성 어려움

---

### 1.4 제어 흐름 그래프 작성법

```
코드:                    그래프:
1: 입력(x)                 [1]
2: if (x > 0)               |
3:   y = x + 1         [2]---조건
4: else                /       \
5:   y = x - 1      [3]참    [5]거짓
6: 출력(y)              \       /
                         [6]
```

- **노드(Node)**: 처리 블록 (문장)
- **간선(Edge)**: 제어 흐름 (화살표)
- **영역(Region)**: 닫힌 영역의 수

**McCabe 순환 복잡도** = 간선(E) - 노드(N) + 2
- 위 예시: E=4, N=4 → 4-4+2 = **2**

---

### 1.5 커버리지 비교표

| 커버리지 | 대상 | 핵심 키워드 | 강도 |
|----------|------|-------------|------|
| **문장** | 모든 문장 | 1번 이상 실행 | 최약 |
| **분기(결정)** | 모든 분기 | 참/거짓 경로 | ↑ |
| **조건** | 각 개별 조건 | 각 조건 T/F | ↑ |
| **조건/결정** | 조건+결정 | 동시 만족 | ↑ |
| **MC/DC** | 독립적 영향 | 각 조건이 독립적으로 영향 | ↑ |
| **다중 조건** | 모든 조합 | 2^N 조합 | ↑ |
| **경로** | 모든 경로 | 전체 실행 경로 | 최강 |

## 2장. 결합도 (Coupling) ★★★

### 2.1 결합도란?

모듈 간의 **상호 의존성** 정도.
**결합도는 낮을수록 좋다!** (독립적 = 유지보수 쉬움)

---

### 2.2 결합도 순서 (약함 → 강함) = 좋음 → 나쁨

```
자료 < 스탬프 < 제어 < 외부 < 공통 < 내용
(약)                                    (강)
좋다 ←――――――――――――――――――――――――――――→ 나쁘다
```

### 암기법: "자스제외공내"

> **자**동차 **스**탬프 **제**조 **외**부 **공**장 **내**부
> 자 - 스 - 제 - 외 - 공 - 내

---

### 2.3 각 결합도 상세 설명

#### (1) 자료 결합도 (Data Coupling) - 가장 약함 = 가장 좋음!
- **단순 데이터(값)**만 매개변수로 전달
- 가장 바람직한 결합

```
예시: calculate(int a, int b)  ← 단순 정수값만 전달
```

#### (2) 스탬프 결합도 (Stamp Coupling)
- **배열, 구조체, 레코드** 등 자료구조를 매개변수로 전달
- 전달받은 자료구조의 일부만 사용할 수도 있음

```
예시: printName(Student student)  ← 구조체 전달 (name만 쓰는데 전체 전달)
```

**시험 키워드**: "배열", "구조체", "자료구조 전달" → **스탬프 결합도**

#### (3) 제어 결합도 (Control Coupling)
- **제어 요소(플래그, 제어 신호)**를 매개변수로 전달
- 호출하는 모듈이 호출당하는 모듈의 **논리를 제어**

```
예시: process(data, flag)
      if (flag == true)
          처리A
      else
          처리B
→ flag로 상대 모듈의 실행 흐름을 제어!
```

**시험 키워드**: "플래그", "제어 신호", "논리 제어" → **제어 결합도**

#### (4) 외부 결합도 (External Coupling)
- 모듈들이 **외부에서 도입된 데이터 포맷, 프로토콜, 장치 인터페이스**를 공유
- 외부 데이터(외부 파일, 외부 장치) 공유

```
예시: 두 모듈이 동일한 외부 파일 형식을 공유하여 사용
```

#### (5) 공통 결합도 (Common Coupling)
- **전역 변수(Global Variable)**를 여러 모듈이 공유
- 한 모듈이 전역 변수를 바꾸면 다른 모듈에 영향

```
예시:
  int globalData = 100;     ← 전역 변수
  moduleA() { globalData++; }
  moduleB() { print(globalData); }
→ A가 바꾸면 B에 영향!
```

**시험 키워드**: "전역 변수", "공유 변수" → **공통 결합도**

#### (6) 내용 결합도 (Content Coupling) - 가장 강함 = 가장 나쁨!
- 다른 모듈의 **내부 기능, 내부 자료를 직접 참조하거나 수정**
- 최악의 결합도, 절대 피해야 함

```
예시: moduleA가 moduleB의 내부 변수를 직접 변경
      moduleA가 moduleB의 중간 지점으로 분기(GOTO)
```

**시험 키워드**: "내부 직접 접근", "내부 참조/수정" → **내용 결합도**

---

### 2.4 결합도 비교표

| 결합도 | 영문 | 전달 대상 | 키워드 | 강도 |
|--------|------|-----------|--------|------|
| **자료** | Data | 단순 데이터 값 | 매개변수(값) | 최약(최선) |
| **스탬프** | Stamp | 배열, 구조체 | 자료구조 전달 | ↑ |
| **제어** | Control | 플래그, 제어신호 | 논리 제어 | ↑ |
| **외부** | External | 외부 데이터 | 외부 공유 | ↑ |
| **공통** | Common | 전역 변수 | 전역 변수 공유 | ↑ |
| **내용** | Content | 내부 직접 접근 | 내부 참조/수정 | 최강(최악) |

## 3장. 응집도 (Cohesion) ★★★

### 3.1 응집도란?

하나의 모듈 내부 요소들이 **서로 관련되어 있는 정도**.
**응집도는 높을수록 좋다!** (하나의 목적 = 유지보수 쉬움)

---

### 3.2 응집도 순서 (약함 → 강함) = 나쁨 → 좋음

```
우연 < 논리 < 시간 < 절차 < 교환(통신) < 순차 < 기능
(약)                                           (강)
나쁘다 ←――――――――――――――――――――――――――――――→ 좋다
```

### 암기법: "우논시절교순기"

> **우**리 **논**문 **시**험 **절**대 **교**수님 **순**서대로 **기**출!
> 우 - 논 - 시 - 절 - 교 - 순 - 기

---

### 3.3 각 응집도 상세 설명

#### (1) 우연적 응집도 (Coincidental) - 가장 약함 = 가장 나쁨!
- 모듈 내 요소들 사이에 **아무런 관계가 없음**
- 그냥 우연히 한 모듈에 모여있음

```
예시: 모듈A {
    print("hello")     ← 출력 기능
    calculate(x, y)    ← 계산 기능
    sendEmail()         ← 이메일 기능
}
→ 전혀 관련 없는 기능들이 모여있음
```

#### (2) 논리적 응집도 (Logical)
- **논리적으로 유사한 기능**을 하나의 모듈에 모아놓음
- 입력/출력/보고서 등 유사한 "종류"의 기능을 모음

```
예시: 모든_출력() {
    프린터_출력()
    화면_출력()
    파일_출력()
}
→ "출력"이라는 논리적 유사성만으로 모아놓음
```

#### (3) 시간적 응집도 (Temporal)
- **특정 시간**에 함께 실행되는 기능들을 모아놓음
- 초기화, 종료 처리 등

```
예시: 프로그램_초기화() {
    변수_초기화()
    파일_열기()
    화면_초기화()
    로그_시작()
}
→ "프로그램 시작 시"라는 같은 시간에 실행
```

**시험 키워드**: "같은 시간", "초기화", "동시에 실행" → **시간적 응집도**

#### (4) 절차적 응집도 (Procedural)
- 모듈 내 요소들이 **순서대로 실행**되어야 하는 관계
- 기능적 관련성보다는 **실행 순서**에 의한 결합

```
예시: 파일_처리() {
    권한_확인()      ← 1단계
    파일_열기()      ← 2단계
    파일_읽기()      ← 3단계
}
→ 순서대로 실행되어야 하지만, 같은 데이터를 공유하지는 않음
```

#### (5) 교환적(통신적) 응집도 (Communication)
- 모듈 내 요소들이 **동일한 입출력 데이터**를 사용
- 같은 데이터를 공유하여 처리

```
예시: 학생_처리(학생_데이터) {
    학생_성적_계산(학생_데이터)
    학생_등급_부여(학생_데이터)
    학생_보고서_출력(학생_데이터)
}
→ 모두 "학생_데이터"라는 동일 데이터를 사용
```

**시험 키워드**: "동일한 데이터 사용", "같은 자료" → **교환적(통신적) 응집도**

#### (6) 순차적 응집도 (Sequential)
- 한 활동의 **출력(결과)**이 다음 활동의 **입력**으로 사용
- 파이프라인 구조

```
예시: 데이터_변환() {
    rawData = 데이터_읽기()          ← 출력: rawData
    cleaned = 데이터_정제(rawData)   ← 입력: rawData, 출력: cleaned
    result = 데이터_분석(cleaned)    ← 입력: cleaned
}
→ 출력 → 입력 → 출력 → 입력 연쇄
```

**시험 키워드**: "출력이 다음 입력", "순차적으로 연결" → **순차적 응집도**

#### (7) 기능적 응집도 (Functional) - 가장 강함 = 가장 좋음!
- 모듈 내 모든 요소가 **단일 기능(하나의 기능)**을 수행하기 위해 존재
- 가장 이상적인 응집도

```
예시: 소득세_계산(급여) {
    과세표준 = 급여 - 공제액
    세율 = 과세표준_세율_결정(과세표준)
    소득세 = 과세표준 * 세율
    return 소득세
}
→ 오직 "소득세 계산"이라는 하나의 기능만 수행
```

---

### 3.4 응집도 비교표

| 응집도 | 영문 | 핵심 키워드 | 강도 |
|--------|------|-------------|------|
| **우연적** | Coincidental | 관계 없는 기능 모음 | 최약(최악) |
| **논리적** | Logical | 유사한 종류 모음 | ↑ |
| **시간적** | Temporal | 같은 시간에 실행 | ↑ |
| **절차적** | Procedural | 순서대로 실행 | ↑ |
| **교환적** | Communication | 동일 데이터 사용 | ↑ |
| **순차적** | Sequential | 출력→입력 연결 | ↑ |
| **기능적** | Functional | 단일 기능 수행 | 최강(최선) |

| 최선 | 자료 결합도 | 기능적 응집도 |
| 최악 | 내용 결합도 | 우연적 응집도 |

## 4장. 디자인 패턴 (Design Pattern) ★★★

### 4.1 GoF 패턴 분류

GoF(Gang of Four) 디자인 패턴은 **23개**이며 3가지로 분류:

| 분류 | 목적 | 대표 패턴 |
|------|------|-----------|
| **생성(Creational)** | 객체 생성 | Singleton, Abstract Factory, Builder, Prototype, Factory Method |
| **구조(Structural)** | 객체 합성/구조 | Adapter, Proxy, Bridge, Composite, Decorator, Facade, Flyweight |
| **행위(Behavioral)** | 객체 상호작용 | Iterator, Visitor, Observer, Command, Strategy, State, Mediator, Chain of Responsibility, Template Method, Memento, Interpreter |

---

### 4.2 시험에 나오는 패턴 집중 정리

#### ★ Singleton (생성 패턴)
- **인스턴스가 오직 하나만** 생성되도록 보장
- 전역적으로 접근 가능한 단일 객체

```
키워드: "하나만", "단일 인스턴스", "유일한 객체"

예시: 프린터 관리자, 데이터베이스 연결 풀
→ 시스템에서 딱 하나만 존재해야 하는 객체
```

#### ★ Abstract Factory (생성 패턴)
- **관련된 객체들의 제품군(family)**을 한꺼번에 생성
- **구체적인 클래스를 지정하지 않고** 인터페이스를 통해 생성
- 별칭: **Kit**

```
키워드: "관련 객체 묶음 생성", "제품군", "Kit", "연관된 객체 집합"

예시: Windows용 UI(버튼, 스크롤바)와 Mac용 UI를 각각 팩토리로 생성
```

#### ★ Adapter (구조 패턴)
- **호환되지 않는 인터페이스**를 변환하여 함께 동작하게 함
- **Wrapper** 패턴이라고도 함

```
키워드: "인터페이스 변환", "호환", "Wrapper", "110V→220V"

예시: 110V 콘센트를 220V로 변환하는 어댑터
     기존 시스템의 인터페이스를 새 시스템에 맞게 변환
```

#### ★ Proxy (구조 패턴)
- **대리 객체**를 통해 실제 객체에 대한 **접근을 제어**
- 실제 객체 대신 대리인이 요청을 처리

```
키워드: "대리", "대리 객체", "접근 제어", "중간 대리인"

예시: 이미지 로딩 프록시 → 실제 이미지 로딩 전까지 대리 객체가 대기
     원격 프록시, 보호 프록시, 가상 프록시
```

#### ★ Iterator (행위 패턴)
- 컬렉션의 내부 구조를 노출하지 않고 **순차적으로 접근**
- hasNext(), next() 등의 메서드 제공

```
키워드: "순차 접근", "반복", "컬렉션 순회", "내부 구조 노출 없이"

예시: 배열, 리스트, 트리 등을 동일한 방법으로 순회
```

#### ★ Visitor (행위 패턴)
- 객체 구조의 **요소에 수행할 연산을 별도 객체로 분리**
- 구조를 변경하지 않고 **새로운 연산을 추가**

```
키워드: "새로운 연산 추가", "객체 구조 방문", "기능 확장"

예시: 파일 시스템의 각 요소(파일, 폴더)에 대해
     → 용량계산 Visitor, 검색 Visitor 등 연산을 추가
```

#### ★ Observer (행위 패턴)
- 객체 상태 변화 시 **의존 객체들에게 자동 통보**
- **1:N 의존 관계**, 발행-구독(Publish-Subscribe) 모델

```
키워드: "상태 변화 통보", "자동 갱신", "1대다", "발행-구독"

예시: 유튜브 채널 구독 → 새 영상 올리면 구독자에게 알림
```

#### ★ Chain of Responsibility (행위 패턴)
- 요청을 처리할 수 있는 객체들을 **체인(사슬)으로 연결**
- 요청이 처리될 때까지 체인을 따라 전달

```
키워드: "체인", "사슬", "요청 전달", "책임 연쇄"
```

#### ★ Command (행위 패턴)
- 요청 자체를 **객체로 캡슐화**
- 실행 취소(Undo), 재실행(Redo) 지원

```
키워드: "요청 캡슐화", "명령 객체화", "Undo/Redo"
```

---

### 4.3 패턴 식별 키워드 정리표

| 키워드가 보이면 | 이 패턴! |
|----------------|----------|
| 하나만, 단일 인스턴스 | **Singleton** |
| 관련 객체 묶음, Kit, 제품군 | **Abstract Factory** |
| 인터페이스 변환, Wrapper, 호환 | **Adapter** |
| 대리 객체, 접근 제어 | **Proxy** |
| 순차 접근, 반복, 순회 | **Iterator** |
| 새 연산 추가, 방문 | **Visitor** |
| 상태 변화 통보, 1:N, 구독 | **Observer** |
| 체인, 사슬, 책임 전달 | **Chain of Responsibility** |
| 요청 캡슐화, Undo | **Command** |

---

### 4.4 패턴 암기 비법

**생성 패턴 5개**: "**싱앱빌프팩**" (Singleton, Abstract Factory, Builder, Prototype, Factory Method)

**구조 패턴 7개**: "**어프브컴데파플**" (Adapter, Proxy, Bridge, Composite, Decorator, Facade, Flyweight)

**행위 패턴**: 나머지 전부 → 그 중 시험 빈출은 **Iterator, Visitor, Observer**

## 5장. UML (Unified Modeling Language)

### 5.1 다이어그램 종류

#### 구조 다이어그램 (정적)
| 다이어그램 | 설명 | 키워드 |
|------------|------|--------|
| **클래스** | 클래스 간 관계 | 속성, 메서드, 관계 |
| **패키지** | 패키지 간 관계 | **폴더 모양** |
| **객체** | 객체 인스턴스 관계 | 특정 시점의 스냅샷 |
| **컴포넌트** | 컴포넌트 간 관계 | 물리적 구성요소 |
| **배치(Deployment)** | 물리적 배치 | 하드웨어 구성 |
| **복합구조** | 내부 구조 | 클래스 내부 |

#### 행위 다이어그램 (동적)
| 다이어그램 | 설명 | 키워드 |
|------------|------|--------|
| **유스케이스** | 사용자-시스템 상호작용 | 액터, 시나리오 |
| **시퀀스** | 시간 순서대로 메시지 교환 | 생명선, 시간 흐름 |
| **활동(Activity)** | 업무 흐름 | 흐름도, 워크플로우 |
| **상태** | 상태 전이 | 이벤트, 상태 변화 |

**시험 포인트**: "패키지 다이어그램의 모양?" → **폴더 모양**

---

### 5.2 클래스 관계

| 관계 | 기호 | 설명 | 예시 |
|------|------|------|------|
| **연관(Association)** | ─── (실선) | 서로 알고 있는 관계 | 학생 ── 수업 |
| **일반화(Generalization)** | ──▷ (빈 삼각형) | 상속 관계 (is-a) | 동물 ◁── 개 |
| **의존(Dependency)** | - - -> (점선 화살표) | 일시적 사용 관계 | 주문 - - -> 결제 |
| **집합(Aggregation)** | ──◇ (빈 마름모) | 부분-전체, 독립 존재 | 팀 ◇── 선수 |
| **합성(Composition)** | ──◆ (채운 마름모) | 부분-전체, 생명주기 동일 | 건물 ◆── 방 |
| **실현(Realization)** | - - -▷ (점선 빈 삼각형) | 인터페이스 구현 | 인터페이스 구현 |

**집합 vs 합성 차이**:
- 집합(Aggregation): 부분이 전체 없이도 **독립적 존재** 가능 (팀이 해체되도 선수는 존재)
- 합성(Composition): 부분이 전체와 **생명주기가 같음** (건물이 없어지면 방도 없어짐)

## 6장. 테스트 기법

### 6.1 스텁(Stub) vs 드라이버(Driver) ★★★

| 구분 | 스텁(Stub) | 드라이버(Driver) |
|------|-----------|-----------------|
| **통합 방식** | **하향식(Top-Down)** | **상향식(Bottom-Up)** |
| **역할** | **하위 모듈** 대체 | **상위 모듈** 대체 |
| **설명** | 아직 개발 안 된 하위 모듈 대신 임시 모듈 제공 | 아직 개발 안 된 상위 모듈 대신 호출해주는 임시 모듈 |

```
하향식 통합 (Top-Down):
  [상위 모듈] ← 테스트 대상
       |
  [스텁(Stub)] ← 하위 모듈 아직 없으니 가짜로 대체!

상향식 통합 (Bottom-Up):
  [드라이버(Driver)] ← 상위 모듈 아직 없으니 가짜로 대체!
       |
  [하위 모듈] ← 테스트 대상
```

**암기법**:
- **스텁 = 하향식 = 하위 대체** → "스**텁**다운(Top-Down), **하**위를 **스**텁"
- **드라이버 = 상향식 = 상위 대체** → "드라이버가 위에서 **운전(Drive)**해줌"

---

### 6.2 동등분할(동치분할, Equivalence Partitioning)

- 입력 데이터를 **유효한 값과 무효한 값으로 분류(그룹화)**
- 각 그룹에서 **대표값 하나만** 테스트
- **블랙박스 테스트** 기법

```
예시: 점수 입력 (0~100점)
  유효 클래스: 0 ≤ x ≤ 100  → 대표값: 50
  무효 클래스1: x < 0        → 대표값: -1
  무효 클래스2: x > 100      → 대표값: 101
```

**경계값 분석(Boundary Value Analysis)**과 함께 출제됨:
- 경계값: 0, 1, 99, 100 (경계 근처 값 테스트)

|----------|------|

추가 구성요소: 테스트케이스 ID, 사전 조건, 사후 조건 등

## 7장. 클라우드 서비스

### IaaS vs PaaS vs SaaS 비교표

| 구분 | IaaS | PaaS | SaaS |
|------|------|------|------|
| **풀네임** | Infrastructure as a Service | Platform as a Service | Software as a Service |
| **제공 범위** | **인프라** (서버, 스토리지, 네트워크) | **플랫폼** (인프라+OS+미들웨어) | **소프트웨어** (완성된 앱) |
| **사용자 관리** | OS, 미들웨어, 앱 | 앱만 개발 | 설정만 |
| **예시** | AWS EC2, Azure VM | Google App Engine, Heroku | Google Docs, Office 365, Dropbox |
| **비유** | 빈 땅 + 건물 임대 | 인테리어된 사무실 임대 | 완성된 서비스 이용 |

**암기법**:
- **I**aaS = **I**nfrastructure = **인프라** (제일 아래)
- **P**aaS = **P**latform = **플랫폼** (중간)
- **S**aaS = **S**oftware = **소프트웨어** (제일 위, 완성품)

```
제공 범위 (아래→위):
  SaaS: [인프라 + OS + 미들웨어 + 애플리케이션] ← 다 해줌
  PaaS: [인프라 + OS + 미들웨어] + 사용자가 앱 개발
  IaaS: [인프라만]              + 사용자가 나머지 다 관리
```

## 8장. CPU 스케줄링 ★★★

### 8.1 선점형 vs 비선점형

| 구분 | 비선점형(Non-preemptive) | 선점형(Preemptive) |
|------|--------------------------|---------------------|
| **정의** | 실행 중인 프로세스를 **중단시킬 수 없음** | 실행 중인 프로세스를 **강제로 중단** 가능 |
| **장점** | 문맥교환 오버헤드 적음 | 응답시간 빠름 |
| **대표** | FCFS, SJF, 우선순위(비선점) | RR, SRT, 우선순위(선점) |

---

### 8.2 주요 스케줄링 알고리즘

#### FCFS (First Come First Served)
- **먼저 도착한 순서대로** 실행
- **비선점형**
- 문제점: 호위 효과(Convoy Effect) - 긴 작업 뒤에 짧은 작업이 대기

#### SJF (Shortest Job First)
- **CPU 버스트가 가장 짧은 것** 우선 실행
- **비선점형**
- 평균 대기시간 최소화에 최적
- 문제점: 기아(Starvation) - 긴 작업이 계속 밀림

#### SRT (Shortest Remaining Time)
- **SJF의 선점형 버전**
- 남은 실행시간이 더 짧은 프로세스가 도착하면 **CPU를 빼앗음**
- = SRTF (Shortest Remaining Time First)

#### RR (Round Robin)
- **타임 퀀텀(Time Quantum)**만큼 돌아가며 실행
- **선점형**
- 타임 퀀텀이 크면 → FCFS에 가까움
- 타임 퀀텀이 작으면 → 문맥교환 오버헤드 증가

---

### 8.3 핵심 용어 정리

| 용어 | 정의 |
|------|------|
| **도착시간(Arrival Time)** | 프로세스가 준비 큐에 도착한 시간 |
| **실행시간(Burst Time)** | CPU를 사용하는 시간 |
| **완료시간(Completion Time)** | 프로세스 실행이 끝난 시간 |
| **반환시간(Turnaround Time)** | 완료시간 - 도착시간 |
| **대기시간(Waiting Time)** | 반환시간 - 실행시간 |

```
대기시간 = 반환시간 - 실행시간
반환시간 = 완료시간 - 도착시간

∴ 대기시간 = 완료시간 - 도착시간 - 실행시간
```

---

### 8.4 평균 대기시간 계산 - 실전 예제 (간트 차트)

#### [예제 1] FCFS 스케줄링

| 프로세스 | 도착시간 | 실행시간 |
|----------|----------|----------|
| P1 | 0 | 6 |
| P2 | 1 | 3 |
| P3 | 2 | 2 |
| P4 | 3 | 4 |

**풀이 과정:**

FCFS = 도착 순서대로 실행 (비선점)

**Step 1: 간트 차트 그리기**
```
|  P1  |  P2 | P3 | P4  |
0      6     9   11    15
```

**Step 2: 각 프로세스 대기시간 계산**

| 프로세스 | 도착 | 실행 | 시작 | 완료 | 반환시간 | 대기시간 |
|----------|------|------|------|------|----------|----------|
| P1 | 0 | 6 | 0 | 6 | 6-0=6 | 6-6=**0** |
| P2 | 1 | 3 | 6 | 9 | 9-1=8 | 8-3=**5** |
| P3 | 2 | 2 | 9 | 11 | 11-2=9 | 9-2=**7** |
| P4 | 3 | 4 | 11 | 15 | 15-3=12 | 12-4=**8** |

**Step 3: 평균 대기시간**
```
평균 대기시간 = (0 + 5 + 7 + 8) / 4 = 20 / 4 = 5.0
```

---

#### [예제 2] SJF 스케줄링 (비선점)

| 프로세스 | 도착시간 | 실행시간 |
|----------|----------|----------|
| P1 | 0 | 6 |
| P2 | 1 | 3 |
| P3 | 2 | 2 |
| P4 | 3 | 4 |

**풀이 과정:**

SJF = 현재 대기 중인 프로세스 중 실행시간이 가장 짧은 것 선택 (비선점)

**Step 1: 실행 순서 결정**
- t=0: P1만 도착 → P1 실행 (실행시간 6)
- t=6: P1 완료. 대기: P2(3), P3(2), P4(4) → **P3 선택 (가장 짧은 2)**
- t=8: P3 완료. 대기: P2(3), P4(4) → **P2 선택 (가장 짧은 3)**
- t=11: P2 완료. 대기: P4(4) → **P4 선택**
- t=15: P4 완료

**Step 2: 간트 차트**
```
|  P1  | P3 |  P2 | P4  |
0      6    8    11    15
```

**Step 3: 대기시간 계산**

| 프로세스 | 도착 | 실행 | 시작 | 완료 | 반환시간 | 대기시간 |
|----------|------|------|------|------|----------|----------|
| P1 | 0 | 6 | 0 | 6 | 6-0=6 | 6-6=**0** |
| P2 | 1 | 3 | 8 | 11 | 11-1=10 | 10-3=**7** |
| P3 | 2 | 2 | 6 | 8 | 8-2=6 | 6-2=**4** |
| P4 | 3 | 4 | 11 | 15 | 15-3=12 | 12-4=**8** |

**Step 4: 평균 대기시간**
```
평균 대기시간 = (0 + 7 + 4 + 8) / 4 = 19 / 4 = 4.75
```

---

#### [예제 3] SRT 스케줄링 (선점형 SJF)

| 프로세스 | 도착시간 | 실행시간 |
|----------|----------|----------|
| P1 | 0 | 8 |
| P2 | 1 | 4 |
| P3 | 2 | 2 |
| P4 | 3 | 1 |

**풀이 과정:**

SRT = 매 시점 남은 실행시간이 가장 짧은 프로세스 실행 (선점!)

**Step 1: 매 시각 상태 추적**

| 시각 | 이벤트 | 대기 프로세스(남은시간) | 실행 |
|------|--------|------------------------|------|
| t=0 | P1 도착 | P1(8) | P1 |
| t=1 | P2 도착 | P1(7), P2(4) | **P2** (4<7, 선점!) |
| t=2 | P3 도착 | P1(7), P2(3), P3(2) | **P3** (2<3, 선점!) |
| t=3 | P4 도착 | P1(7), P2(3), P3(1), P4(1) | P3 계속 (1=1, 현재 실행 유지) |
| t=4 | P3 완료 | P1(7), P2(3), P4(1) | **P4** (1이 최소) |
| t=5 | P4 완료 | P1(7), P2(3) | **P2** (3<7) |
| t=8 | P2 완료 | P1(7) | **P1** |
| t=15 | P1 완료 | - | 끝 |

**Step 2: 간트 차트**
```
|P1|  P2 | P3 |P4|  P2 |    P1     |
0  1     2    4  5     8          15
```
(주의: t=2에서 P3가 선점하므로 P2는 t=2에서 중단, t=5에서 재개)

수정된 간트 차트:
```
|P1| P2 |P3|P4| P2  |    P1      |
0  1    2  4  5     8           15
```

**Step 3: 대기시간 계산**

| 프로세스 | 도착 | 실행 | 완료 | 반환시간 | 대기시간 |
|----------|------|------|------|----------|----------|
| P1 | 0 | 8 | 15 | 15-0=15 | 15-8=**7** |
| P2 | 1 | 4 | 8 | 8-1=7 | 7-4=**3** |
| P3 | 2 | 2 | 4 | 4-2=2 | 2-2=**0** |
| P4 | 3 | 1 | 5 | 5-3=2 | 2-1=**1** |

**Step 4: 평균 대기시간**
```
평균 대기시간 = (7 + 3 + 0 + 1) / 4 = 11 / 4 = 2.75
```

#### [예제 4] 라운드 로빈(RR) 스케줄링

| 프로세스 | 도착시간 | 실행시간 |
|----------|----------|----------|
| P1 | 0 | 6 |
| P2 | 1 | 3 |
| P3 | 2 | 2 |

**타임 퀀텀 = 2**

**풀이 과정:**

RR = 각 프로세스를 타임 퀀텀(2)만큼 돌아가며 실행

**Step 1: 실행 순서 추적**

| 시각 | 실행 | 남은시간 | 큐 상태 |
|------|------|----------|---------|
| t=0~2 | P1 (2만큼) | P1:4 | [P2, P3, P1] (P2는 t=1 도착, P3는 t=2 도착) |
| t=2~4 | P2 (2만큼) | P2:1 | [P3, P1, P2] |
| t=4~6 | P3 (2만큼) | P3:0 완료! | [P1, P2] |
| t=6~8 | P1 (2만큼) | P1:2 | [P2, P1] |
| t=8~9 | P2 (1만큼) | P2:0 완료! | [P1] |
| t=9~11 | P1 (2만큼) | P1:0 완료! | [] |

**Step 2: 간트 차트**
```
| P1 | P2 | P3 | P1 |P2| P1 |
0    2    4    6    8  9   11
```

**Step 3: 대기시간 계산**

| 프로세스 | 도착 | 실행 | 완료 | 반환시간 | 대기시간 |
|----------|------|------|------|----------|----------|
| P1 | 0 | 6 | 11 | 11-0=11 | 11-6=**5** |
| P2 | 1 | 3 | 9 | 9-1=8 | 8-3=**5** |
| P3 | 2 | 2 | 6 | 6-2=4 | 4-2=**2** |

**Step 4: 평균 대기시간**
```
평균 대기시간 = (5 + 5 + 2) / 3 = 12 / 3 = 4.0
```

## 9장. 페이지 교체 알고리즘 ★★★

### 9.1 페이지 교체란?

물리 메모리(프레임)가 가득 찼을 때, 새 페이지를 적재하기 위해 **어떤 페이지를 내보낼지** 결정하는 알고리즘.

**페이지 부재(Page Fault)**: 필요한 페이지가 메모리에 없을 때 발생.

---

### 9.2 주요 알고리즘

| 알고리즘 | 교체 대상 | 핵심 키워드 |
|----------|-----------|-------------|
| **FIFO** | **가장 먼저 들어온** 페이지 | 선입선출, 가장 오래된 |
| **LRU** | **가장 오래 사용하지 않은** 페이지 | 최근 최소 사용 |
| **LFU** | **가장 적게 사용된** 페이지 | 사용 빈도 최소 |
| **OPT** | **앞으로 가장 오래 사용하지 않을** 페이지 | 최적(이론적) |

**구분 포인트**:
- FIFO: "들어온 **시점**" 기준 (가장 **먼저 적재**된 것)
- LRU: "사용한 **시점**" 기준 (가장 **오래전에 참조**된 것)
- LFU: "사용 **횟수**" 기준 (가장 **적게 참조**된 것)

---

### 9.3 페이지 부재 횟수 계산 - 실전 예제 (프레임 표 그리기)

#### [예제 1] FIFO (프레임 3개)

**참조 문자열**: 1, 2, 3, 4, 1, 2, 5, 1, 2, 3

**풀이 과정:**

FIFO = 가장 먼저 들어온 페이지를 교체

| 참조 | 프레임1 | 프레임2 | 프레임3 | 부재? | 교체 설명 |
|------|---------|---------|---------|-------|-----------|
| **1** | **1** | - | - | O (Fault) | 빈 프레임에 적재 |
| **2** | 1 | **2** | - | O (Fault) | 빈 프레임에 적재 |
| **3** | 1 | 2 | **3** | O (Fault) | 빈 프레임에 적재 |
| **4** | **4** | 2 | 3 | O (Fault) | 1이 가장 먼저 들어옴 → 1 교체 |
| **1** | 4 | **1** | 3 | O (Fault) | 2가 가장 먼저 들어옴 → 2 교체 |
| **2** | 4 | 1 | **2** | O (Fault) | 3이 가장 먼저 들어옴 → 3 교체 |
| **5** | **5** | 1 | 2 | O (Fault) | 4가 가장 먼저 들어옴 → 4 교체 |
| **1** | 5 | 1 | 2 | X (Hit) | 1이 이미 있음 |
| **2** | 5 | 1 | 2 | X (Hit) | 2가 이미 있음 |
| **3** | 5 | **3** | 2 | O (Fault) | 1이 가장 먼저 들어옴 → 1 교체 |

**페이지 부재 횟수 = 8회**

---

#### [예제 2] LRU (프레임 3개)

**참조 문자열**: 1, 2, 3, 4, 1, 2, 5, 1, 2, 3

**풀이 과정:**

LRU = 가장 오래 **사용하지 않은** 페이지를 교체 (최근 참조 시점이 가장 오래된 것)

| 참조 | 프레임 상태 | 부재? | 교체 설명 |
|------|-------------|-------|-----------|
| **1** | {1} | O | 빈 프레임에 적재 |
| **2** | {1, 2} | O | 빈 프레임에 적재 |
| **3** | {1, 2, 3} | O | 빈 프레임에 적재 |
| **4** | {**4**, 2, 3} | O | 1이 가장 오래전 사용 → 1 교체 |
| **1** | {4, **1**, 3} | O | 2가 가장 오래전 사용(t=2) → 2 교체 |
| **2** | {4, 1, **2**} | O | 3이 가장 오래전 사용(t=3) → 3 교체 |
| **5** | {**5**, 1, 2} | O | 4가 가장 오래전 사용(t=4) → 4 교체 |
| **1** | {5, 1, 2} | X | 1이 이미 있음 (최근 사용시점 갱신) |
| **2** | {5, 1, 2} | X | 2가 이미 있음 (최근 사용시점 갱신) |
| **3** | {**3**, 1, 2} | O | 5가 가장 오래전 사용(t=7) → 5 교체 |

**페이지 부재 횟수 = 8회**

> 주의: LRU에서 Hit가 발생하면 해당 페이지의 **최근 사용 시점이 갱신**된다!

---

#### [예제 3] LRU (프레임 4개) - 더 쉬운 예제

**참조 문자열**: 1, 2, 3, 2, 1, 5, 2, 3, 5, 1

**프레임 수: 4개**

| 순서 | 참조 | F1 | F2 | F3 | F4 | 부재? | 설명 |
|------|------|----|----|----|----|-------|------|
| 1 | **1** | 1 | - | - | - | O | 빈 프레임 |
| 2 | **2** | 1 | 2 | - | - | O | 빈 프레임 |
| 3 | **3** | 1 | 2 | 3 | - | O | 빈 프레임 |
| 4 | **2** | 1 | 2 | 3 | - | X | Hit (2 있음, 사용시점 갱신) |
| 5 | **1** | 1 | 2 | 3 | - | X | Hit (1 있음, 사용시점 갱신) |
| 6 | **5** | 1 | 2 | 3 | 5 | O | 빈 프레임 |
| 7 | **2** | 1 | 2 | 3 | 5 | X | Hit |
| 8 | **3** | 1 | 2 | 3 | 5 | X | Hit |
| 9 | **5** | 1 | 2 | 3 | 5 | X | Hit |
| 10 | **1** | 1 | 2 | 3 | 5 | X | Hit |

**페이지 부재 횟수 = 4회** (프레임이 많으면 부재가 적어짐)

---

### 9.4 FIFO의 특이 현상: Belady의 모순

- **프레임 수를 늘렸는데** 오히려 **페이지 부재가 증가**하는 현상
- FIFO에서만 발생할 수 있음
- LRU, OPT에서는 발생하지 않음 (스택 알고리즘)

### 10.2 chmod - 파일 권한 변경 ★★★

#### 권한 구조

```
    소유자(Owner)  그룹(Group)  기타(Others)
       rwx           rwx          rwx
```

| 문자 | 의미 | 숫자 값 |
|------|------|---------|
| **r** | 읽기(Read) | **4** |
| **w** | 쓰기(Write) | **2** |
| **x** | 실행(Execute) | **1** |
| - | 권한 없음 | 0 |

#### 숫자 권한 계산법

```
rwx = 4 + 2 + 1 = 7
rw- = 4 + 2 + 0 = 6
r-x = 4 + 0 + 1 = 5
r-- = 4 + 0 + 0 = 4
--x = 0 + 0 + 1 = 1
--- = 0 + 0 + 0 = 0
```

#### 자주 나오는 권한 조합

| 명령 | 소유자 | 그룹 | 기타 | 의미 |
|------|--------|------|------|------|
| `chmod 755` | rwx(7) | r-x(5) | r-x(5) | 소유자 전체, 나머지 읽기+실행 |
| `chmod 644` | rw-(6) | r--(4) | r--(4) | 소유자 읽기쓰기, 나머지 읽기만 |
| `chmod 700` | rwx(7) | ---(0) | ---(0) | 소유자만 전체 권한 |
| `chmod 777` | rwx(7) | rwx(7) | rwx(7) | 모든 사용자 전체 권한 |

#### 시험 출제 유형

```
문제: chmod 751 file.txt 에서 기타 사용자(Others)의 권한은?

풀이: 751 → 소유자(7) 그룹(5) 기타(1)
      기타 = 1 = --x = 실행 권한만 있음

답: 실행(Execute)
```

```
문제: 소유자는 읽기/쓰기/실행, 그룹은 읽기/실행, 기타는 실행만 가능한 chmod 명령은?

풀이: 소유자 = rwx = 4+2+1 = 7
      그룹   = r-x = 4+0+1 = 5
      기타   = --x = 0+0+1 = 1

답: chmod 751
```

## 11장. 실전 예상 문제 10선

---

### [문제 1] 테스트 커버리지

> 프로그램의 조건문 `if (A > 0 AND B < 5)` 에서 각 개별 조건식이 참(True)과 거짓(False)을 모두 한 번 이상 갖도록 테스트하는 커버리지는?

**정답**: 조건 커버리지 (Condition Coverage)

**해설**: 각 "개별 조건"(A>0, B<5)이 T/F를 모두 갖는 것 = 조건 커버리지

---

### [문제 2] 결합도

> 모듈 A가 모듈 B를 호출할 때, 처리 방법을 결정하는 플래그(flag)를 매개변수로 전달하는 경우의 결합도는?

**정답**: 제어 결합도 (Control Coupling)

**해설**: 플래그, 제어 신호 전달 = 제어 결합도

---

### [문제 3] 응집도

> 모듈 내의 기능들이 동일한 입력/출력 데이터를 공유하며 서로 다른 처리를 수행하는 경우의 응집도는?

**정답**: 교환적(통신적) 응집도 (Communication Cohesion)

**해설**: 동일한 데이터를 사용 = 교환적 응집도

---

### [문제 4] 디자인 패턴

> 호환되지 않는 인터페이스를 가진 클래스들이 함께 동작할 수 있도록 인터페이스를 변환해주는 디자인 패턴은?

**정답**: 어댑터 패턴 (Adapter Pattern)

**해설**: 인터페이스 변환, 호환 = Adapter (Wrapper)

---

### [문제 5] 디자인 패턴

> 클래스의 인스턴스가 오직 하나만 생성되는 것을 보장하고, 이에 대한 전역적인 접근점을 제공하는 디자인 패턴은?

**정답**: 싱글톤 패턴 (Singleton Pattern)

---

### [문제 6] 테스트

> 하향식(Top-Down) 통합 테스트에서 아직 개발되지 않은 하위 모듈을 대체하는 임시 모듈은?

**정답**: 스텁 (Stub)

**해설**: 하향식 = 스텁 (하위 모듈 대체) / 상향식 = 드라이버 (상위 모듈 대체)

### [문제 7] CPU 스케줄링 계산

> 다음 프로세스들을 SJF(비선점) 스케줄링할 때 평균 대기시간은?
>
> | 프로세스 | 도착시간 | 실행시간 |
> |----------|----------|----------|
> | P1 | 0 | 3 |
> | P2 | 1 | 6 |
> | P3 | 2 | 1 |
> | P4 | 3 | 4 |

**풀이**:
- t=0: P1(3) 실행
- t=3: P1 완료. 대기: P2(6), P3(1), P4(4) → **P3(1)** 선택
- t=4: P3 완료. 대기: P2(6), P4(4) → **P4(4)** 선택
- t=8: P4 완료. 대기: P2(6) → **P2(6)** 선택
- t=14: P2 완료

```
간트 차트:
| P1 |P3|  P4  |    P2    |
0    3  4      8         14
```

| 프로세스 | 도착 | 실행 | 완료 | 대기시간 |
|----------|------|------|------|----------|
| P1 | 0 | 3 | 3 | 3-0-3=**0** |
| P2 | 1 | 6 | 14 | 14-1-6=**7** |
| P3 | 2 | 1 | 4 | 4-2-1=**1** |
| P4 | 3 | 4 | 8 | 8-3-4=**1** |

**정답**: 평균 대기시간 = (0+7+1+1)/4 = **9/4 = 2.25**

### [문제 8] 페이지 교체 계산

> 페이지 참조 문자열이 2, 3, 1, 2, 5, 2, 3, 5 이고 프레임이 3개일 때, FIFO 알고리즘의 페이지 부재 횟수는?

**풀이**:

| 참조 | F1 | F2 | F3 | 부재? | 설명 |
|------|----|----|-----|-------|------|
| 2 | 2 | - | - | O | 빈 프레임 |
| 3 | 2 | 3 | - | O | 빈 프레임 |
| 1 | 2 | 3 | 1 | O | 빈 프레임 |
| 2 | 2 | 3 | 1 | X | Hit |
| 5 | 5 | 3 | 1 | O | 2가 가장 먼저 → 교체 |
| 2 | 5 | 2 | 1 | O | 3이 가장 먼저 → 교체 |
| 3 | 5 | 2 | 3 | O | 1이 가장 먼저 → 교체 |
| 5 | 5 | 2 | 3 | X | Hit |

**정답**: 페이지 부재 횟수 = **6회**

### [문제 9] chmod

> `chmod 764 file.txt` 명령에서 그룹(Group) 사용자의 권한을 모두 쓰시오.

**풀이**: 764 → 소유자(7) 그룹(**6**) 기타(4)
- 6 = 4 + 2 = rw-

**정답**: 읽기(Read), 쓰기(Write)

### [문제 10] 클라우드 서비스

> 클라우드 서비스 모델 중, 서버, 스토리지, 네트워크 등의 인프라를 가상화하여 제공하고, 사용자가 운영체제와 애플리케이션을 직접 관리하는 서비스 모델은?

**정답**: IaaS (Infrastructure as a Service)
